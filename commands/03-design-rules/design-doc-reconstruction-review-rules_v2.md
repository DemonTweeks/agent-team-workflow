## 目标

- 对**单份或批量**设计返构文档（V6 结构）执行**完整性**、**规范性**、**图表正确性**、**代码一致性**与**证据可追溯性**检查。
- 以 **「每个章节 = 独立检查单元」** 为粒度逐项核对。
- 输出**通过/不通过**结论、**问题清单**（阻断/主要/建议）与**可执行修改建议**。

### 文档流水线中的位置

- **上游**：`design-doc-reconstruction-rules_v2.md` 产出设计文档（V6 结构）；依据来自后端代码（Controller、Service、Domain、Repository、Mapper、SQL、ORM 实体）及前端代码。
- **交叉引用**：可对照 `01-接口文档.md` 检查方法标识与接口契约的一致性。
- **下游**：定稿设计文档作为架构评审、技术债务分析、重构规划的输入。

## 一、文档级检查（必检）

| 项 | 要求 | 不满足时 |
|----|------|----------|
| 标题 | `# {模块/功能名称} — 开发设计文档` | **主要** |
| 适用范围 | 明确功能点描述 | **主要** |
| 基线分支 | 明确分支名称 | **建议** |
| 章节顺序 | 严格按 0-9 章顺序 | **阻断** |
| 需求映射（第0章） | 存在表格（需求编号、PRD章节、需求描述、对应设计章节） | **主要** |
| 代码增量清单（第1章） | 按服务分组，文件级变更清单 | **主要** |
| 数据模型（第2章） | 按服务分组，DTO/实体/表结构定义 | **主要** |
| 接口设计（第3章） | 按服务分组，接口契约 | **主要** |
| 业务规则（第4章） | 全局规则，编号化 | **阻断** |
| 核心链路（第5章） | 按链路分组，Mermaid 图 | **阻断** |
| 前端规格（第6章） | 如适用，组件架构 | **建议** |
| 枚举配置（第7章） | 全局枚举/常量/国际化 | **主要** |
| 测试策略（第8章） | 全局测试场景 | **建议** |
| 需求映射表（第9章） | 全局汇总，实现状态 | **主要** |

## 六、第4章：业务规则与校验检查

## 十五、严重级别

| 级别 | 含义 | 示例 |
|------|------|------|
| 阻断 | 无法作为设计参考 | 章节缺失、规则无编号、代码增量与 git 不一致、调用链与代码不符 |
| 主要 | 影响完整性与一致性 | 缺子节、缺表、DTO 与代码不一致、规则无错误码、需求未映射 |

## 十六、输出格式（执行本命令时必须）

1. **结论**：通过 / 不通过（列阻断、主要、建议数量）。
2. **文档级检查结果**：章节完整性检查。
3. **按章节结果**：第0-9章逐章检查。
4. **图表质量小结**：Mermaid 图问题。
5. **证据可追溯性检查**：锚点、置信度、冲突。
6. **与代码一致性差异**：如提供代码。
7. **问题清单表**：编号、级别、章节、描述、期望修复、证据锚点。
