## 流水线与范围对账

- **上游**：`feature-list-generation-rules_v1.md` → **`功能清单.md`**；**`prd-generation_v8.md`**（含 **§9.1** 准入 DoD 自检）→ **`{功能点名称}_业务需求文档.md`**。
- **本规则**：每份 BRD 对应 **`prd-review/{功能点名称}_产品需求完整性检查.md`**（**含 §2.19** 与 **`xuqiuceshidod.md`** 对账及 **测试准入结论**）。
- **下游**：**`api-doc-generation-rules_v1.md`** → **`api-doc-review-rules_v1.md`**；批量 BRD 稳定后 **`prd-feature-map-generation.md`** 生成图谱；**测试设计** 以 **`xuqiuceshidod.md`** 为准入规则、以本检查 **§2.19** 为 BRD 侧执行落点。

**可选加强（不替代正文既有检查项）**：当用户同时提供 **`功能清单.md`** 且被检 BRD 可映射到清单中某一 **`###` 模块** 时，在检查报告 **检查概览** 中增加 **「与功能清单覆盖对账」**：将该模块已枚举的 **查询项、列表列、操作、表单分区、审批** 等与 BRD 的 **`## 功能需求`**、**`## 界面布局和交互`**、**`## 业务流程`** 等逐类核对；**清单已列出而 BRD 未落档** 且未进入 **`## 不明确条目`** → **错误** 或 **严重错误**（依是否构成验收缺口判定）。未提供清单时注明 **「未执行功能清单覆盖对账」**。

## 指令说明

基于业务需求生成指令（**`prd-generation_v8.md`**，含 **BRD-i 实现颗粒度标准**、**第 10 节来源与缺口**、**第 11.5～11.6 节深度清单**、**第 12 节返构输出质量**、**第 12.7 节 BRD↔PRD 冲突对账**、**第 12.8 节代码返构专项门禁**）的要求，对生成的业务需求文档进行全面检查，确保文档完整、准确、符合规范。检查重点关注：

1. **章节完整性**：是否包含所有 **20** 个标准章节（**含 `## 不明确条目`**）
2. **章节顺序**：是否严格按照标准文档结构模板的顺序
3. **指令遵循度**：是否严格遵循 `prd-generation_v8.md` 的各项要求（含流程图、业务语义、Schema 等）
4. **BRD-i 实现颗粒度**：是否具备字段中英名、数据项表、操作确认/成功/失败提示语（含英文列或说明）、校验规则矩阵、验收 Given/When/Then、**事件流/异常/准入/权限与角色（§1a～1e、§11.7 含相关服务与权限矩阵）** 等与 **BRD-i 同级范例文档**（见 **2.13** 说明，路径以仓库内实际范例为准）的可交付细节（允许因无界面/无通知而合理删减，但须在文档中显式标注「不涉及」）
5. **复杂审批 / 校验域深度**（条件强制，见 **2.15**）：涉及审批中心、多分支流程、会签/区域会签/备案、邮件与移动审批、国际或多业态提交校验、附件与合规等时，是否达到 **`prd-generation_v8.md` 第 11.5～11.6 节** 所列最低落档；声明依据 PRD 时是否执行 **第 12.7 节** 对账项
6. **代码返构专项门禁**（条件强制，见 **2.16**）：文档 **主要来自代码反构** 且 **存在 P0 PRD 或声明与 PRD 对齐** 时，是否逐项落实 **`prd-generation_v8.md` 第 12.8 节** 门禁 **G-01～G-11**（概述信息密度、查询导出、审批路径同源、审批人表、会签清单、邮件与 SRM、移动与智能风险、提交校验全量、运维删除与终态、驳回/状态语义、附录承诺等）；未落实项是否进入开放问题，**禁止**以「结构完整」代替「与 PRD 可对齐」
7. **业务语义完整度和准确度**：业务语义表达是否完整、准确；在**无独立产品 PRD 时**是否准确反映代码逻辑；在**存在已评审产品 PRD 或需求概述中声明依据 PRD 时**，是否与之对齐（见 **第三步补充：双源与优先级**）
8. **代码业务逻辑完整表达验证**（核心要求，必须严格执行）：严格验证所有代码的业务逻辑是否完整表达，不能遗漏任何业务逻辑分支、异常处理、边界条件等
9. **内容相关性验证**（核心要求，必须严格执行）：严格验证输出的内容是否与要分析的内容是强相关，不能包含与功能点无关的内容，不能添加超出代码范围的内容
10. **基于代码分析验证**（核心要求，必须严格执行）：严格验证输出内容**可追溯**；凡在**需求概述/文档说明**中声明依据 PRD 的条款，**允许以 PRD 为真源**（须注明出处或引用），**不**视为「无代码依据」；其余条款须能追溯到代码实现或明确 **TODO/待确认**，禁止无依据编造（**与第 3 步补充条款一致**）
11. **子功能点代码结合剖析（§1f）**（核心要求，与 `prd-generation_v8.md` **§1f** 对齐）：`## 功能需求` 是否已下沉到 **子功能点/原子操作** 颗粒度；每个子功能点是否具备 **实现逻辑、规则挂接、流程与时序、调用与数据流、依赖、代码追溯与技术组件印证**；规则是否在功能需求与 **`## 业务规则`** 间 **可引用**；**禁止** 仅有功能域标题而无剖析表或等价分条（详见 **第二步 2.17**）
12. **业务术语表与子功能点详细业务逻辑（§1.0）**（核心要求，与 `prd-generation_v8.md` **§1.0** 对齐）：`## 需求概述`（或 **`## 文档说明`** / **`## 业务概述`** 的等价位置）是否具备可检索的 **`### 业务术语表`**（或 **`### 术语`**），且 **术语+业务含义** 覆盖 **§1.0.1** 范围；每个子功能点是否具备 **§1.0.2** 要求的 **详细业务逻辑说明**（主路径、分支、上下游联动、异常、规则挂接），**禁止** 仅剖析表占位或一句话概括（详见 **第一步 1.5**、**第二步 2.18**）
13. **需求测试准入 DoD**（**必须执行**，与 **`xuqiuceshidod.md`**、**`prd-generation_v8.md` §9.1** 对齐）：按 **第二步 2.19** 核对 **红线 R1～R7**、**必要条件 N1～N12**、**建议项 S1～S8**；输出 **测试准入结论** 及 **与 BRD 完整性检查总体结论的合并说明**（红线触发时 **完整性检查不得单独判为「通过」** 而不注明不准入，见 **2.19**）

**检查报告输出**：
- 检查完成后，将生成完整的检查报告
- **报告保存路径**：与业务需求文档相同目录下的`prd-review`子目录，文件名为`{功能点名称}_产品需求完整性检查.md`
- 功能点名称从被检查的业务需求文档路径中自动提取
- 如果 `prd-review` 子目录不存在，将自动创建
- 示例：若 BRD 路径为 `{BRD_ROOT}/样机管理/合同管理/样机合同申请单_业务需求文档.md`，则检查报告保存路径为 `{BRD_ROOT}/样机管理/合同管理/prd-review/样机合同申请单_产品需求完整性检查.md`（`prd-review` 与 BRD 同目录）

#### 1.3 章节标题规范性检查

**检查项**：章节标题必须使用标准章节标题，不能随意修改

**检查方法**：检查章节标题是否与标准标题一致，不一致标记为**错误**

#### 1.4 非标准章节检查

**检查项**：**标准主体章节**固定为 **`prd-generation_v8.md` 所列 20 章**（含 **`## 不明确条目`**）；禁止在 **20 章之外**再增加其他 **`##` 级「业务章节」**（如另起「待确认清单」重复承载第 20 章内容）。**`## 文档说明`**、**`## 业务概述`** 等若存在，须符合 **11.1** 约定且**不**替代第 20 章。

**检查方法**：检查是否存在 **未定义的 `##` 主体章节**；检查 **`## 不明确条目`** 是否缺失或标题错误；违规标记为**错误**或**严重错误**

#### 1.5 业务术语表完整性检查（与 `prd-generation_v8.md` §1.0.1、§12.4 对齐）

**检查项**：

- **存在性**：在 **`## 需求概述`** 内须存在 **`### 业务术语表`**（推荐）；若术语表仅在 **`## 文档说明`** 或 **`## 业务概述`** 中，须仍有 **明确三级标题**（如 **`### 术语`** / **`### 业务术语表`**），**禁止**正文散落而无集中表。
- **表格结构**：至少包含 **术语/名称**、**业务含义**（独立定义，禁止大量「见上文」）、**典型使用场景或范围**；**推荐**列 **英文或缩写**、**出处（PRD 章节 / 推断依据）**（与 **§1.0.1** 一致）。
- **覆盖度**：术语表须覆盖本 BRD 中 **功能需求、业务规则、业务流程** 出现的 **核心业务对象、状态/阶段专名、审批/会签专名、关键外部系统名、主要角色称谓** 等；**仅 2～3 行**且无状态/审批等专名、或明显遗漏 PRD 已定义词条 → **错误**；完全缺失术语表 → **严重错误**。
- **P0 对齐**：声明依据 PRD 时，术语 **应与 PRD 术语表同词**；与 PRD 冲突而无双写说明 → **严重错误**（与 **第三步 场景 B** 联动）。
- **未定义专名**：后文多次出现的重要专名在术语表中 **无定义、无 PRD 引用、未进开放问题/不明确条目** → **错误**。

**检查方法**：检索 `## 需求概述`（及文档说明/业务概述）中的术语小节；抽样对照 **`## 功能需求`** 前三页与 **`## 业务规则`** 中的专名是否均有表项或脚注。

### 第二步：指令遵循度检查

#### 2.1 模块归属信息完整性检查

**检查项**：模块归属信息章节必须包含以下内容：
- 所属模块：业务模块名称
- 所属子模块层级：逐层列出所有子模块（如有）
- 功能点名称：具体功能点
- 完整模块路径：`模块名/子模块1/子模块2/.../功能点名称`

**检查方法**：检查必需字段和模块路径格式，缺失或格式错误标记为**错误**

#### 2.2 业务入口清单完整性检查

**检查项**：业务入口清单章节必须包含：
- 主要业务入口：列出所有主要的业务入口（JOB、API接口、前端功能等）
- **定时任务清单**（核心要求，必须包含）：
  - 列出所有与功能点相关的定时任务（JOB）
  - 每个定时任务的业务含义和使用场景
  - 每个定时任务的执行时机和频率
  - 每个定时任务处理的业务数据范围
  - 每个定时任务的业务影响和业务价值
- **内嵌页面清单**（核心要求，必须包含）：
  - 列出所有内嵌页面（弹窗、子窗体、用户控件等）
  - 每个内嵌页面的业务含义和使用场景
- **逐层打开页面清单**（核心要求，必须包含）：
  - 列出所有逐层打开的页面（详情页面、编辑页面、查询页面等）
  - 每个打开页面的业务含义和使用场景
  - 页面打开关系和层级结构说明
- **从主功能入口触发的关联功能入口清单**（核心要求，必须严格执行）：
  - 列出所有从主功能入口（如待处理列表页面、查询列表页面等）触发的关联功能入口
  - 每个关联功能入口的业务含义和使用场景
  - 每个关联功能入口的触发位置和触发条件

#### 2.13 实现颗粒度与文案完整性检查（BRD-i）（核心要求，与 `prd-generation_v8.md` 对齐）

**依据**：业务需求生成指令 **`prd-generation_v8.md`** 中的 **「实现颗粒度标准（BRD-i）」**；对照 **BRD-i 同级范例文档**（字段中英名、操作/错误提示、校验矩阵、验收 GWT、通知/移动等颗粒度）。范例路径以**当前仓库/项目约定**为准（如 `docs/generated/acceptance/工作量确认_业务需求文档new.md`）；**无范例文件时**以 **`prd-generation_v8.md` 内嵌 BRD-i 样例片段** 为最低对照标准。

**检查项**（凡本功能点确有界面、列表、导出、审批、通知等能力者必须满足；**确无对应能力**时须在相关标准章节内**显式写明「本功能点不涉及…」**，不得留空白不交代）：

#### 2.17 子功能点代码结合剖析完整性检查（与 `prd-generation_v8.md` §1f 对齐，核心要求，必须严格执行）

**依据**：**`prd-generation_v8.md`** → **§1f 子功能点 / 原子操作 / 业务规则的代码结合剖析**；与 **§9** 映射表中 **「1f 子功能点代码结合剖析」** 行一致。

**检查项**：

1. **拆解粒度**：`## 功能需求` 是否按 **业务功能域** 下设 **子功能点**（或 BR-xx 下三级结构），且子功能点下有 **原子操作** 列举（界面操作 + 非界面入口：JOB、回调、独立 API 等）；**禁止** 仅有一句「本模块支持查询、申请、审批」类概括而无子结构。
2. **子功能点剖析维度**：每个子功能点是否至少具备 **§1f** 所列维度中的可验收落档——**标识与范围**、**实现逻辑（业务语义，含分支/批处理语义）**、**规则与判定**（引用业务规则表或编号）、**流程与时序**（或指向 **`## 业务流程`** / **`## 界面操作时序`** 的图）、**调用与数据流（业务侧）**、**代码与配置追溯**（与 **`## 技术组件清单`** 可交叉核对）、**依赖**、**异常与第 20 章**；整块缺失 → **严重错误**。
3. **原子操作覆盖**：每个用户可见操作（含列表行操作、批量、导出、弹窗确认路径）是否在子功能点下 **分条**；代码/界面存在而文档合并为「可执行相关操作」→ **错误**；主路径遗漏 → **严重错误**。
4. **规则挂接**：`## 功能需求` 中的校验/权限/状态描述是否 **引用** **`## 业务规则`** / **`## 前端校验规则`** 中编号或表行；**规则悬浮**（规则章有表、功能需求从不引用）→ **错误**。
5. **三处同源**：同一操作的 **功能需求剖析**、**`## 业务逻辑链路`**、**`## 技术组件清单`** 是否 **无矛盾**（入口、存储过程/API、外部系统一致）；两处以上冲突 → **严重错误**。
6. **辅助操作 / JOB / 回调**：`## 辅助操作业务逻辑分析`、**定时任务**、**外部回调** 是否在 **`## 功能需求`** 有 **§1f** 级交叉引用或摘要；完全缺失交叉 → **错误**。
7. **双源（场景 B）**：PRD 子场景与代码子路径不一致时，是否 **并列说明** 或 **开放问题**，**禁止** 仅写实现侧而无 PRD 对账说明（与 **2.14** 联动）。

**检查方法**：通读 `## 功能需求`，按上表 **7** 项勾选；与 **2.5（功能需求完整性）** 合并判定时：**2.17** 未通过则 **2.5** 不得单独判为通过。在检查报告中可单列 **「§1f 子功能点剖析」** 小节。**确无多子场景** 的极简功能点，须在 `## 功能需求` **显式声明**「本子功能仅含单一路径」并仍给出 **单个子功能点剖析表**，否则按 **拆解粒度** 缺失处理。

#### 2.18 子功能点详细业务逻辑深度检查（与 `prd-generation_v8.md` §1.0.2 对齐，核心要求）

**依据**：**`prd-generation_v8.md`** → **§1.0.2 各子功能点详细业务逻辑**；与 **§1f 剖析表**、**`## 业务流程`** / **`## 业务逻辑链路`** 同源校验。

#### 3.1 代码业务逻辑完整表达验证（核心要求，必须严格执行）

**检查项**：严格验证所有代码的业务逻辑是否完整表达，不能遗漏任何业务逻辑分支、异常处理、边界条件等

**完整表达要求**：
- **所有业务逻辑分支必须完整表达**：
  - 必须包含所有if-else分支、switch-case分支、循环逻辑等
  - 必须包含所有异常处理逻辑（try-catch、异常分支处理等）
  - 必须包含所有边界条件处理（空值检查、范围检查、状态检查等）
  - 必须包含所有数据校验逻辑（前端校验、后端校验、数据库约束等）
- **所有业务规则必须完整表达**：
  - 必须包含代码中实现的所有业务规则，不能遗漏
  - 必须包含所有条件判断逻辑（状态判断、权限判断、数据判断等）
  - 必须包含所有业务约束（唯一性约束、关联性约束、业务限制等）
- **所有数据操作必须完整表达**：
  - 必须包含所有数据查询逻辑（查询条件、查询范围、查询结果处理等）
  - 必须包含所有数据更新逻辑（更新条件、更新字段、更新后处理等）
  - 必须包含所有数据插入逻辑（插入条件、插入字段、插入后处理等）
  - 必须包含所有数据删除逻辑（删除条件、删除方式、删除后处理等）
- **所有外部接口调用必须完整表达**：
  - 必须包含所有外部接口调用的完整链路（参数拼装、接口调用、结果处理）
  - 必须包含所有异常处理和重试机制
  - 必须包含所有回调处理逻辑

**检查方法**：**代码对比检查**（核心要求）：对于文档中描述的每个业务逻辑，必须能追溯到具体代码实现；检查代码中是否存在文档未描述的业务逻辑分支、异常处理逻辑、边界条件处理，存在标记为**严重错误**。**场景 B 补充**：若文档声明 PRD 依据，代码中未实现或分支与 PRD 不一致时，在报告中记为 **差异项**（**警告** 或 **待确认**），**不**与场景 A 同等视为「文档未覆盖代码」**严重错误**（**除非**文档已明确声称实现范围与 PRD 完全一致且可核验）。

#### 3.2 内容相关性验证（核心要求，必须严格执行）

**检查项**：严格验证输出的内容是否与要分析的内容是强相关，不能包含与功能点无关的内容，不能添加超出代码范围的内容

**相关性要求**：
- **内容必须与功能点强相关**：
  - 所有描述的业务功能必须属于该功能点的业务范围
  - 所有描述的业务规则必须与该功能点直接相关
  - 所有描述的数据操作必须与该功能点直接相关
  - 不能包含与该功能点无关的业务描述
- **内容不能超出代码范围**（**与「第三步补充」场景 A/B 联动**）：
  - **场景 A（仅代码反构）**：所有描述的业务逻辑、规则、字段须在代码中有明确实现或引用；不能添加代码中不存在的业务逻辑、业务规则、字段等
  - **场景 B（声明 PRD 依据）**：允许**引用 PRD 原文**作为业务条款来源；**不要求**每条 PRD 条款在代码中已有实现（可能未开发或版本差异），但文档须标注依据或待确认；**禁止**将其他功能点或通用常识与 PRD 混写而不标注来源
- **内容必须基于实际代码**（场景 A）或 **PRD+代码可追溯**（场景 B）：
  - 场景 A：所有描述应能追溯到具体代码实现；不能基于推测或假设添加内容；不能基于其他功能点的代码添加内容；不能基于通用业务知识添加内容
  - 场景 B：PRD 条款以引用为准；**其余**条款仍应能追溯到代码或标 **TODO**

**检查方法**：**相关性验证**（核心要求）：检查文档描述的业务功能是否属于该功能点业务范围，是否存在不相关内容，存在标记为**错误**。**代码范围验证**（核心要求）：在**场景 A** 下，检查文档描述的每个业务逻辑、业务规则、字段是否在代码中有明确实现，不存在标记为**严重错误**；在**场景 B** 下，PRD 引用条款**不**单独做「代码不存在」判失败。**代码追溯验证**：场景 A 下无法追溯标记为**严重错误**；场景 B 下对 PRD 条款检查是否**标明出处**，未标明且与功能无关则**错误**

#### 3.3 基于代码分析验证（核心要求，必须严格执行）

**检查项**：严格验证输出的内容是严格基于代码分析得到，而不是推测或者假设，所有业务规则、流程、字段来源等必须有明确的代码依据

**基于代码分析要求**（**默认场景 A**；若文档为 **场景 B**（声明 PRD 依据），见下段「双源例外」）：

- **所有业务规则必须有代码依据**：
  - 每条业务规则必须能够追溯到具体的代码实现（方法、类、文件等）
  - 每条业务规则必须明确说明代码中的判断逻辑（查询哪个表、使用哪个字段、使用什么条件判断）
  - 不能基于方法名或注释推测业务规则，必须查看代码的实际实现逻辑
  - 不能基于业务常识推测业务规则，必须基于代码的实际逻辑
- **所有业务流程必须有代码依据**：
  - 每个业务流程步骤必须能够追溯到具体的代码实现
  - 每个业务流程分支必须能够追溯到具体的代码判断逻辑
  - 不能基于推测描述业务流程，必须基于代码的实际执行路径
- **所有字段数据来源必须有代码依据**：
  - 每个字段的数据来源必须能够追溯到具体的代码实现
  - 每个字段的数据获取逻辑必须明确说明代码中的实现方式
  - 不能基于推测描述字段来源，必须基于代码的实际实现
- **所有业务逻辑必须有代码依据**：
  - 每个业务逻辑必须能够追溯到具体的代码实现
  - 每个业务逻辑的执行路径必须明确说明代码中的实现方式
  - 不能基于推测描述业务逻辑，必须基于代码的实际实现

**双源例外（场景 B）**：对**明确标注来自 PRD 的条款**，验证重点为「是否**忠实转写**、是否与 **2.14** 信息源优先级一致」；**不**仅因代码未实现而判**严重错误**，应标 **警告/待确认**。**未标注来源**却与代码不符的条款，仍按**推测**处理。

**检查方法**：**代码依据验证**（核心要求）：检查文档描述的每个业务规则、业务流程、字段来源、业务逻辑是否能追溯到具体代码实现，无法追溯标记为**严重错误**。**推测内容识别**（核心要求）：检查是否存在基于方法名、注释、业务常识推测的描述，存在标记为**严重错误**。**代码逻辑验证**：检查业务规则、业务流程、字段来源是否明确说明代码中的实际判断逻辑、执行路径、获取方式，不明确或与代码逻辑不一致标记为**错误**。**场景 B 叠加**：PRD 条款不适用「代码依据」单条判失败规则，按 **第三步补充** 与 **2.14** 处理

#### 3.4 业务语义转换检查（核心要求，必须严格执行）

**检查项**：文档必须严格遵循业务语义转换规则，禁止使用技术术语

**技术术语禁止清单**：
- 禁止使用SQL技术术语：WHERE、EXISTS、UNION、JOIN、SELECT、FROM等
- 禁止使用技术术语：Controller、Service、Mapper、VO、PO等（应使用业务术语）
- 禁止直接使用SQL语句或SQL技术术语描述业务逻辑

**业务语义表达要求**：
- 所有校验逻辑、业务规则、数据查询逻辑都必须用业务语言描述
- 使用"业务筛选条件"替代"WHERE条件"

- 使用"多业务对象合并检查"替代"UNION查询"
- 使用"业务对象关联关系"替代"JOIN关联条件"
- 使用"状态为'已接收'"替代"STATUS='RECEIVED'"

**检查方法**：检查是否使用SQL技术术语（WHERE、EXISTS、UNION、JOIN、SELECT、FROM等）或技术术语（Controller、Service、Mapper、VO、PO等），使用标记为**严重错误**

#### 3.5 业务规则完整度检查（核心要求，必须严格执行）

**检查项**：业务规则必须完整、准确，不能简化描述或推测

**完整度要求**：
- **明确校验对象**：每条业务规则必须明确说明校验的业务对象（如：出库单号、箱子号、位置序列号等），不能混淆不同的业务对象
- **明确校验范围**：每条业务规则必须明确说明校验的范围（如：所有出库单，还是只检查立库仓库的出库单），不能遗漏范围限制
- **完整校验条件**：每条业务规则必须完整说明校验的具体条件（必须用业务语义描述，禁止使用SQL技术术语），包括：
  - 业务筛选条件（如：状态为"已接收"）
  - 业务对象存在性检查的逻辑（如：检查箱子是否存在于其他出库任务单中）
  - 多业务对象合并检查的逻辑（如：检查箱子是否存在于出库任务单或位置移动单中）
  - 业务对象关联关系（如：通过出库单ID关联出库任务单和明细）
  - 所有涉及的数据表和业务字段（必须包含Schema前缀）
- **不能简化描述**：不能仅根据方法名或注释来简化描述业务规则，必须完整说明校验的完整逻辑
  - 不能仅说"重复接收校验"，必须说明检查的是什么（是检查出库单号，还是检查箱子号）
  - 不能仅说"存在正在处理的单据"，必须说明是什么单据（是其他出库任务单，还是位置移动单），以及检查的条件是什么
- **术语与代码保持一致**：业务规则中的术语必须与代码中的实际表达保持一致，不能自己创造术语
- **明确代码判断逻辑**：如果业务规则涉及某种判断（如某种仓库类型、某种状态等），必须明确说明代码中实际的判断逻辑（如查询哪个表、使用哪个字段、使用什么条件判断），而不是仅使用一个术语

**检查方法**：检查每条业务规则是否明确校验对象、校验范围、完整校验条件（用业务语义描述）、数据表和字段信息（包含Schema前缀），是否简化或推测，术语是否与代码实际表达一致，不完整或不准确标记为**错误**

#### 3.6 字段数据来源完整度检查（核心要求，必须严格执行）

**检查项**：字段数据来源必须完整、准确，不能遗漏任何字段

**完整度要求**：
- **所有字段的数据来源都必须明确记录**，不能遗漏任何字段
- **前端交互数据来源**：
  - 所有通过弹窗查询获取的字段必须明确说明弹窗功能、接口和数据来源
  - 所有通过下拉框、选择框获取的字段必须明确说明数据接口和来源
  - 所有通过搜索功能获取的字段必须明确说明搜索接口和数据来源
  - 所有通过数据联动获取的字段必须明确说明联动规则和数据来源
- **自动计算字段**：
  - 所有自动计算的字段必须明确说明计算公式、计算逻辑和计算涉及的字段
  - 计算公式和计算逻辑必须详细说明，确保可以复现
- **默认值字段**：
  - 所有有默认值的字段必须明确说明默认值的来源和设置规则
- **后端数据来源**：
  - 所有通过接口参数获取的字段必须明确说明参数的来源和传递路径
  - 所有通过数据库查询获取的字段必须明确说明查询的存储过程（**必须包含Schema前缀**）或数据查询逻辑（必须用业务语义描述，不能直接使用SQL语句）、查询涉及的数据表（**必须包含Schema前缀**）、查询条件和业务规则
  - 所有通过外部系统接口获取的字段必须明确说明外部接口的调用方式和参数、外部接口返回数据的处理逻辑、外部接口调用的业务规则
- **数据流转路径**：
  - 每个字段从数据源头到最终使用的完整流转路径必须清晰完整
  - 包括所有中间环节（前端交互、接口调用、数据处理等）
  - 确保开发者可以根据文档完整实现字段的数据获取逻辑

**检查方法**：检查是否所有字段的数据来源都已明确记录，前端交互数据来源、自动计算字段、默认值字段、后端数据来源是否完整。**Schema前缀检查**（核心要求）：检查所有存储过程和数据表是否包含Schema前缀，缺少标记为**严重错误**。**业务语义表达检查**（核心要求）：检查数据库查询逻辑是否使用SQL技术术语，使用标记为**严重错误**

#### 3.7 外部接口调用链路完整度检查（核心要求，必须严格执行）

**检查项**：外部接口调用链路必须追踪到最底层实现，不能只停留在"调用接口"这一层

**完整度要求**：
- **参数拼装链路必须完整**：
  - 必须说明参数的数据来源（从哪些业务数据中提取）
  - 必须说明参数的数据转换逻辑（如何转换、如何计算）
  - 必须说明参数拼装的业务规则（哪些字段需要拼装、拼装的格式要求、必填/可选字段的判断）
  - 必须说明参数校验逻辑（参数格式校验、参数范围校验、参数必填校验等）
  - 记录完整的参数拼装链路：`业务数据 -> 数据提取 -> 数据转换 -> 参数拼装 -> 参数校验 -> 接口调用`
- **接口调用实现链路必须完整**：
  - 必须说明接口调用的具体实现类和方法
  - 必须说明接口调用的技术实现（HTTP调用、WebService调用、消息队列等）
  - 必须说明接口调用的配置信息（接口地址、超时设置、重试机制等）
  - 必须说明接口调用的异常处理逻辑（网络异常、超时异常、业务异常等）
  - 记录完整的接口调用链路：`参数准备 -> 接口调用实现 -> 异常处理 -> 结果返回`
- **结果处理链路必须完整**：
  - 必须说明接口返回结果的解析逻辑（如何解析返回数据、数据格式转换等）
  - 必须说明结果数据的业务处理（如何更新业务状态、如何保存结果数据等）
  - 必须说明结果数据的校验逻辑（结果有效性校验、结果完整性校验等）
  - 必须说明异常结果的处理逻辑（校验失败的处理、异常情况的处理等）
  - 必须说明结果数据的流转路径（结果数据如何更新到业务对象、如何影响业务流程等）
  - 记录完整的结果处理链路：`接口返回 -> 结果解析 -> 结果校验 -> 业务处理 -> 状态更新`
- **异步回调处理链路必须完整**（如果涉及异步回调）：
  - 必须说明回调接口的实现逻辑（回调接口的接收、参数解析、业务处理）
  - 必须说明回调触发的业务逻辑（状态更新、流程流转、数据同步等）
  - 必须说明回调异常处理（回调失败的重试机制、超时处理、异常通知等）
  - 必须说明回调结果的持久化（回调结果如何保存、如何更新业务状态等）
  - 记录完整的回调处理链路：`回调接收 -> 参数解析 -> 业务处理 -> 状态更新 -> 结果持久化`

**检查方法**：如果提到外部系统集成，检查是否包含参数拼装链路、接口调用实现链路、结果处理链路、异步回调处理链路（如涉及）；只停留在"调用接口"这一层标记为**严重错误**

#### 3.8 代码逻辑与业务语义一致性检查（核心要求，必须严格执行）

**检查项**：业务语义表达必须准确反映代码逻辑，不能与代码逻辑不一致

**一致性要求**：
- **术语与代码保持一致**：业务规则中的术语必须与代码中的实际表达保持一致，不能自己创造术语
- **从代码逻辑推断术语**：如果代码中没有明确的业务术语（如某个业务概念），必须通过代码逻辑推断出准确的业务含义，而不是使用前端代码中的术语或自己创造术语
- **明确代码判断逻辑**：如果业务规则涉及某种判断（如某种仓库类型、某种状态等），必须明确说明代码中实际的判断逻辑（如查询哪个表、使用哪个字段、使用什么条件判断），而不是仅使用一个术语
- **避免使用前端术语**：不要仅使用前端代码中的术语（如变量名、控件名等），必须查看后端代码和数据库代码中的实际判断逻辑
- **代码优先原则**：当代码中的表达与注释、前端术语不一致时，必须以代码中的实际逻辑为准
- **完整代码链路**：对于业务规则中的术语，必须追踪完整的代码链路（前端->后端->数据库），确保术语在所有层级都一致，如果不一致，必须明确说明每个层级的判断逻辑

**检查方法**：检查业务规则中的术语是否与代码实际表达一致，是否明确说明代码中实际的判断逻辑（查询哪个表、使用哪个字段、使用什么条件判断），是否仅使用前端术语而未查看后端和数据库代码，不一致标记为**错误**

- **突出重点**：对于严重错误，要突出显示，但不要重复描述
- **避免冗余信息**：不要在报告中重复指令中的检查标准和要求，只报告检查结果
