# 需求准入测试 DoD（Definition of Done）

> **适用范围**：产品需求文档（PRD）进入测试环节前的准入检查。本仓库中由 **`prd-generation_v8.md`** 生成的 **`{功能点名称}_业务需求文档.md`（BRD，标准 20 章）** 在纳入测试设计前 **须** 满足本 DoD；生成阶段可对照 **`prd-generation_v8.md` →「实现颗粒度」→ §9.1** 自检。
> **检查目标**：内容是否完整、清晰、可测，确保AI生成用例和人工测试设计不返工。
> **结论类型**：通过 / 有条件通过 / 不准入。
> **执行落点**：BRD 完成后使用 **`prd-completeness-check.md`** 做完整性检查，其中 **§2.19** 按本文 **R / N / S** 逐条对账并输出与本表 **同构** 的 **测试准入结论**（通过 / 有条件通过 / 不准入）；**结论以本文规则为准**，§2.19 为检查侧执行说明与证据汇总。

---

## 与标准 BRD（`prd-generation_v8`）章节映射（便于产品/测试对档）

以下仅说明 **主要落档位置**，细节仍以 **`prd-generation_v8.md` BRD-i 与 §9.1** 为准。

| DoD 维度 | 红线/条件 | BRD 中主要对应章节（`##`） |
| :--- | :--- | :--- |
| 业务规则与状态 | R1、N2 | `业务流程`、`业务规则`、`功能需求`（事件流 §1a/§1f） |
| 前置与造数 | R2 | `功能需求`、`需求概述`（边界、角色、预置数据） |
| 事件流与交互 | R3、N4、N5 | `界面操作时序`、`界面布局和交互`、`业务流程` |
| 权限与数据范围 | R4、N11 | `功能需求`、`业务规则`（与 prd-generation **§11.7**） |
| 字段与校验 | R5、N3 | `数据需求`、`前端校验规则`、`业务规则`、功能域数据项表 |
| 数据来源与联动 | R6 | `字段数据来源`、`业务逻辑链路` |
| 可读与可解析 | R7 | 全文；**禁止**关键需求仅图片 |
| 功能边界 | N1 | `功能需求`（子功能点 §1f）、`模块归属信息` |
| 一致性与术语 | N6、N12 | `需求概述`（**业务术语表 §1.0**）、全文与表 |
| 变更波及 | N7 | `需求概述`、`业务依赖关系` |
| 外部依赖 | N8 | `业务逻辑链路`、`业务依赖关系` |
| 导入导出与批量 | N9 | `功能需求`、`业务规则`、`非功能需求`（若有） |
| 错提示与终态 | N10 | `业务规则`、`前端校验规则`、`使用场景`/验收 GWT |
| 建议项 S5 等 | S1～S8 | `非功能需求`、`使用场景`、`需求概述` 等 |

**说明**：BRD **第 20 章 `## 不明确条目`** 中登记的内容，在准入上通常对应 **「有条件通过」** 中须澄清的项，测试用例相关章节应标记 **BLOCKED**（与第二节表下说明一致）。

## 一、红线规则（一票否决 · 触及任意一条即不准入）

| 编号 | 规则项 | 判定标准（满足任一即触发） |
| :--- | :--- | :--- |
| **R1** | **核心业务规则 / 状态流转完全缺失或极度模糊** | 审批怎么过、驳回变什么状态、什么条件能编辑/删除，完全没有文字描述或流程图；或只有一句无法落地的概括（如"按正常流程审批"） |
| **R2** | **前置条件模糊到无法构造测试数据** | 只写"用户已登录"，未写明角色、权限、系统需预置哪些基础数据（如至少存在1个有效项目）；导致测试无法准备环境和数据 |
| **R3** | **事件流歧义严重，AI/测试必然理解错误** | 一句话塞N个动作；存在隐性条件未明示（典型：不写"点击查询按钮"，默认理解为"进入页面即加载数据"）；扩展/异常流缺失导致分支场景无法覆盖 |
| **R4** | **权限边界完全缺失** | 未定义：哪些角色能操作、能看哪些数据（行级/列级范围）、跨部门/跨租户是否隔离；必然漏掉越权测试 |
| **R5** | **关键字段约束 / 数据校验完全缺失** | 核心业务字段（如金额、数量、状态、编号）的长度、格式、必填/非必填、唯一性、枚举值、超长截断还是拦截，全部未定义 |
| **R6** | **核心数据源 / 取数逻辑未说明或无法追溯** | 跨系统接口、跨模块查询、页面间数据联动、下拉框动态数据来源**未说明**；或**仅给出编码、缩写、内部术语，但未提供其业务含义、数据范围和查询路径**（如"快码知识库 COMP-002"未解释编码含义及如何获取具体数据） |
| **R7** | **存在无法解析的内容** | 关键需求仅用纯图片/截图表达无文字、存在乱码、包含`<>`等前端语法字符，导致AI/测试无法读取有效需求 |

**红线结论**：若需求文档触发任意一条红线，**直接判定为不准入**，必须打回产品补全后重新提交。

## 二、必要条件（必须澄清 · 缺失则判定为"有条件通过"）

必要条件不满足时，需求可进入测试设计，但必须同步产出**《需求澄清清单》**，由产品限期补全。未补全前，测试用例对应章节标记为 BLOCKED。

| 编号 | 规则项 | 通过标准 |
| :--- | :--- | :--- |
| **N1** | **功能点拆分清晰，边界不耦合** | 一个需求文档不揉杂N个独立功能；每个功能点有明确边界，输入/输出/状态不互相牵连 |
| **N2** | **事件流完整** | 基本流（主成功场景）+ 扩展流（分支变化）+ 异常流（错误处理）齐全；每步独立，一步一动作，一步一系统响应 |
| **N3** | **字段约束完整可断言（含边界值）** | 所有可输入字段明确定义：长度限制、格式要求（正则）、必填/可选、默认值、枚举值列表、超长/超限时截断还是报错；硬上限必须量化（如会签人数≤10、重付次数≤3、批量上限50条、字符串长度边界值验证） |
| **N4** | **表格规范（AI 可识别）** | 禁止使用合并单元格；核心业务字段不允许留空（无默认值须显式标注"无"）；禁止用"同上/同左"等省略词；相同类型组件使用统一术语（输入框/下拉框/单选框/模糊搜索） |
| **N5** | **组件类型与描述一致** | 文字描述、表格定义、原型图/Demo三处保持一致；严禁表格写"下拉框"实际是"模糊搜索" |
| **N6** | **一致性规范（名称·逻辑·关联）** | 名称一致：同一概念全文术语统一，不混用不同名词指代同一事物；逻辑一致：文档前后描述不自相矛盾（条件、规则、范围不冲突）；文表关联：文字描述与表格数据、流程逻辑建立对应关系，不脱节 |
| **N7** | **变更需求必须写明波及范围** | 明确：波及哪些特性/功能点、具体变更点、影响的数据表/接口/权限、测试回归范围 |
| **N8** | **外部接口 / 依赖明确** | 调用的外部系统接口须说明：协议类型、接口地址或Topic、请求/响应关键字段、超时时间、重试机制、异常/降级处理逻辑 |
| **N9** | **导入 / 导出 / 上传规则详尽（含分页批量上限）** | 文件格式（xlsx/csv/pdf）、大小上限、文件名长度限制、同步还是异步、校验规则（字段映射/空值/重复/格式校验）；列表默认分页大小、最大分页大小；批量操作（提交/审核/作废）单次上限；导出最大支持数据量；超出后是分批还是拒绝 |
| **N10** | **错误提示 & 最终状态明确** | 每种校验失败/异常/驳回场景，系统提示语原文是什么；操作后的单据/数据最终状态是什么（可验证、可断言） |
| **N11** | **数据权限规则（行级 / 列级）** | 明确数据范围过滤逻辑：按组织/租户/部门/项目/供应商等维度隔离；跨范围查看/操作是否需要特殊权限 |
| **N12** | **核心概念与术语无歧义** | 关键业务术语（如"未生效"、"正常数据"、"到货款"）必须定义其业务含义和范围；禁止使用"正常"、"适当"、"相关"、"尽快"等无法量化的模糊词修饰核心规则；同一表述在文档各处只能有一种合理解释 |

## 三、建议项（不影响准入 · 有更好，没有也能测）

| 编号 | 规则项 | 说明 |
| :--- | :--- | :--- |
| **S1** | 需求背景、来源、TCO过程描述 | 有助于理解业务动机，但不影响用例设计 |
| **S2** | 验收准则使用 Given-When-Then 格式 | 如用例能同步生成，此项可省略 |
| **S3** | 流程图、原型图、状态流转图 | 复杂流程强烈建议补充，简单功能可豁免 |
| **S4** | UI 元素位置、展示时机描述 | 如"列表上方常驻提示语"、"鼠标悬停显示Tooltip" |
| **S5** | 非功能性需求 | 性能指标（≤3秒）、并发数、兼容性、多语言、安全防护 |
| **S6** | 信息去重、扁平化组织 | 避免相同内容重复编写；信息组织方式便于阅读和检索 |
| **S7** | 版本变更记录完整 | 便于追溯，但测试只认当前最终态 |
| **S8** | 链接标注用途 | 标注"可忽略"或"本次需求实现必须使用本链接内容作为输入" |

## 四、准入结论判定标准

| 结论 | 判定条件 | 后续动作 |
| :--- | :--- | :--- |
| **通过** | 红线：0条触发<br>必要条件：全部满足 | 直接进入测试设计环节 |
| **有条件通过** | 红线：0条触发<br>必要条件：有缺失 | 产出《需求澄清清单》，产品限期补全；测试可先行设计已通过部分用例，缺失部分标记 BLOCKED |
| **不准入** | 红线：≥1条触发 | 直接打回产品，必须补全红线项后重新提交准入检查 |

## 五、Skill 输出格式建议

准入检查报告建议按以下结构输出，与《预付款管理准入检查报告》保持一致：

1. **业务理解**：业务目标、核心范围、关键流程、最复杂风险点
2. **规则检查结果**：红线(7条) / 必要条件(12条) / 建议项(8条) → 逐条标记【通过 / 不通过 / 缺失 / 不适用】
3. **模糊点清单**：逐条列出 → 问题点 + 影响范围 + 建议方案
4. **波及范围评估**：前端 / 后端 / 数据库 / 外部接口 / 定时任务 / 权限（如文档信息足够）
5. **高风险点推导**：基于业务规则自动推导并发、幂等、异步、数据一致性等技术风险（仅提示，不影响准入结论）
6. **准入结论**：通过 / 有条件通过 / 不准入，附理由和优先级建议
