## 1. 目标

1. **特性分层**：按业务价值链/能力域划分产品域（5～10 个），每域挂载若干模块（文件夹）。
2. **功能点清单（模块摘要）**：主输出模板 **§7**（图谱文档内章节序号）可保留「模块 → 功能需求子节标题串」；**全量** 必须为 **`## 功能需求`** 下 **每个 `###` 子节** 一行（见 **本文 §4.1 附录**）。*旧称 `### 2.x` 仅当 BRD 正文确实采用数字小节时沿用，否则以实际 `###` 标题为锚点。*
3. **三元组建模（核心）**：所有跨模块关系 **必须** 表示为 **(主体, 谓词, 客体)**，并附 **证据**（文档路径 + 章节，如 `` `## 业务依赖关系` > 功能点间依赖 ``；旧模板可写 `§4.1 上游` 但须在 **§9 开放问题** 注明「非 v8 标准章」）。
4. **功能点级关系（v3）**：在附录 **文档×文档边表** 中给出 **功能级锚点**（`M-<模块文件夹名>-<子节标识>`，`子节标识` 与 BRD 中 **`###` 标题** 或团队约定的 **FR 编号** 一致），避免仅写模块名导致无法追溯。
5. **图谱输出**：由三元组 **派生** Mermaid 图；**禁止**先画图再反填表（避免方向错误）。
6. **外部系统**：客体可为 `EXT:系统名`；不入文档—文档边，单独成表。

## 2. 三元组定义

### 2.1 主体 (Subject)

- 本仓库 **一个 PRD 模块**，使用 **文件夹名**（如 `yanshouguanli`），必要时附中文名。
- **禁止**一个三元组主体同时绑定两个模块；若 BRD/PRD 成对，拆成两行并注释关系。

### 2.2 客体 (Object)

- **内部模块**：另一文件夹名。
- **外部系统**：`EXT:ISCP`、`EXT:ERP`、`EXT:FOL`、`EXT:邮件中心` 等，名称与 PRD 表格一致。
- **抽象消费方**：仅在 PRD 写「各业务模块」时使用 `EXT:各业务模块`（或拆成显式枚举，若 PRD 有列举）。

### 2.3 谓词 (Predicate) — 受控词表

| 谓词代码 | 中文表述 | 含义 | 典型读法 |
|----------|----------|------|----------|
| `DEPENDS_DATA` | 数据依赖 | 主体业务需要客体提供的主数据/单据数据 | 客体 → 主体（数据先就绪） |
| `DEPENDS_PLATFORM` | 平台依赖 | 主体依赖客体提供流程/引擎/待办能力 | 客体 → 主体 |
| `PRODUCES_FOR` | 产出/触发 | 主体在业务规则下生成或触发客体侧单据/状态 | 主体 → 客体 |
| `CONSTRAINS` | 校验/约束 | 主体在提交/审批时校验客体状态或受客体规则限制 | 客体 → 主体（ gate ） |
| `UPDATES_STATE_OF` | 回写状态 | 主体操作导致客体实体状态变更 | 主体 → 客体 |
| `INFLUENCES_ROLE` | 角色影响 | 主体配置影响客体模块中角色/审批人判定（非主数据流） | 主体 ⇢ 客体（虚关系） |
| `AGGREGATES_FROM` | 聚合归档 | 主体从客体拉取数据形成归档视图 | 客体 → 主体 |
| `OBSERVES` | 观测/重试 | 横切能力观测或重试客体 | 客体 → 主体（日志消费方） |

**方向规则**：

- `DEPENDS_*`：**客体是前提**，图示中箭头 **从客体指向主体**（`O --> S`），表示「没有/未就绪则主体无法继续」。
- `PRODUCES_FOR`：**主体产生下游**，箭头 **从主体指向客体**（`S --> O`）。
- `CONSTRAINS`：与 `DEPENDS_DATA` 类似，强调校验场景；若 PRD 写「校验是否存在在途 X」，则 **X 模块是客体**，**当前模块是主体**。

**易错点（必须自检）**：

- 「A 审批通过后 **自动生成** B」→ `(A, PRODUCES_FOR, B)`，**不是** `B DEPENDS A` 作为上游数据依赖。
- 「B **基于** A 创建」→ `(B, DEPENDS_DATA, A)`，箭头 **A --> B**。
- 「归档/报表 **引用** PR、PO、发票」→ `(归档模块, AGGREGATES_FROM, pr|paigongguanli|fapiaoguanli)`，箭头 **源单据 → 归档**（**不是** `归档 --> PR`）。
- 「**取消/变更** 时 **校验在途** 工作量确认」→ 校验 **消费** 在途单据状态，箭头 **yanshouguanli --> pobianggengguanli**（与 **`## 业务依赖关系`** 表「主体=变更单」一致时，**CONSTRAINS** 行客体为被校验模块）。
- 「项目外包配置 **影响** 新派结付 / PO / 发票」→ 主体必须是 **xiangmuwaibaopeihi**，客体为被影响模块（**不要**写反）。

## 3. 抽取步骤

### 3.0 与 `prd-generation_v8.md` 标准 BRD 的章节映射（必读）

| 概念 | **v8 标准 20 章 BRD**（真源） | 历史/其它模板（须显式标注） |
|------|------------------------------|-----------------------------|
| 功能点细则 | **`## 功能需求`** 及其下 **`### …`**（子功能域、BR-xx、子功能点剖析块等） | 旧版 **`## 2 功能需求`**、`### 2.x` → 抽取时 **等价于** v8 的 `###` 子节，证据列写真实标题 |
| 模块级依赖 | **`## 业务依赖关系`**（第 18 章）：功能点间、外部系统、数据表等 | 旧版 **§4.1 上游 / §4.2 下游** → 仅作别名；**不得**与 v8 的「第 4 章」混淆（v8 第 4 章语义为 **`## 业务规则`**） |

**执行原则**：优先读取 **`prd-generation_v8.md`** 规定的 **标准章节标题**；若文件结构不符，在 **§9 开放问题** 记录「模板版本」，仍完成三元组但证据列用 **实际 `##` 标题**。

### 第一步：锁定主 PRD 文件

- 通配：`*业务需求文档.md`、`*产品需求文档.md`（以仓库约定为准）；默认排除 `prd-review/**`、`*完整性检查*`。

### 第二步：逐文档填两张表（模块级）

对每个模块 BRD/PRD：

1. **上游类依赖**（数据/平台/校验门槛）→ 从 **`## 业务依赖关系`** 中「上游、前置、依赖的外部模块/系统」等 **表格或分节** 抽取，转为三元组，谓词多为 `DEPENDS_DATA` / `DEPENDS_PLATFORM` / `CONSTRAINS`。*若正文只有混排一张总表，按表行抽取，证据统一标 **`## 业务依赖关系`**。*
2. **下游类影响**（产出、状态回写、角色影响、聚合）→ 从 **`## 业务依赖关系`** 中「下游、被影响、触发」等分节或同表中的类型列抽取，谓词多为 `PRODUCES_FOR` / `UPDATES_STATE_OF` / `INFLUENCES_ROLE` / `AGGREGATES_FROM`。

若同一联系在「上游」与「下游」表述中 **语义重复**，合并为一行，证据列可列两处子节标题。

*兼容旧模板*：若文档 **仅有** `§4.1`/`§4.2` 而无标准 **`## 业务依赖关系`**，仍按原节号抽取，并在 **§9** 标注 **「待迁移至 v8 第 18 章结构」**。

### 第二步 B：功能点级下沉（v3）

在 **主文档 §3（全量三元组表）定稿后**，对每个 BRD 的 **`## 功能需求` 下每个 `###` 子节**：

1. 生成一行 **附录 B** 表：功能 ID（`M-<模块>-<子节标识>`）、模块、子节标识、**`###` 标题原文**、关联内部 PRD 文件夹、谓词、`ROLLUP` 或简述。
2. 若该子节在正文明确点名另一模块或 **`## 业务依赖关系`** 已覆盖模块级边，在 **附录 C** 增加 **带功能锚点** 的边（源 `M-a-<子节>` → 目标 `M-b-<子节>` 或 `(*)` 表示整文档/**`## 业务依赖关系`** 汇总）。
3. 纯列表/导出/本模块 CRUD 且无跨文档点名 → **关联** 列可填 `—`，**关系** 填 `ROLLUP`。

### 第三步：模块别名映射

维护「PRD 用语 → 文件夹名」对照（示例）：

| PRD 用语 | 文件夹名 |
|----------|----------|
| PR需求管理 / 采购申请管理 | `xuqiuguanli`（通用 PR）；新派结付场景另见 `xinpaijiefupr` |
| 派工单管理 / PO派工单 | `paigongguanli` |
| 工作量管理 / 工作量确认 / 工作量申请单 | `yanshouguanli`（以各 PRD 术语为准） |
| 预接收单 / 预收单 | `yishoudanguanli` |
| 审批中心 / 审批流引擎 | `shenpizhongxin`（引擎若单独拆分需在开放问题注明） |

### 第四步：由三元组生成 Mermaid

- 仅 **内部模块** 参与「文档—文档图」。
- 边方向与谓词一致；边上用 **简短中文** 标注谓词类型（如 `数据依赖`、`触发`）。
- 边过多时：拆 **价值链图**（仅 `PRODUCES_FOR` 主链）与 **全量依赖图**（分域 subgraph）。

### 第五步：质量门禁

- [ ] 每条三元组有 **证据**（路径 + §）。
- [ ] 无证据的边 **不得** 出现。
- [ ] 与「易错点」清单对照，重点检查 **验收/预收/发票** 链方向。
- [ ] 开放问题：**`## 业务依赖关系`**（或旧 §4）表格明显把「下游」误写入「上游」时，**照抄原文并标注疑似章节错误**。

## 4. 输出 Markdown 模板

```markdown
# {产品名}：产品特性与 PRD 三元组图谱

## 0. 元数据与范围
## 1. 产品域划分
## 2. 谓词与符号说明
## 3. 全量三元组表（按模块分组）
## 4. 外部系统三元组（客体以 EXT: 开头）
## 5. 派生视图：价值链（Mermaid）
## 6. 派生视图：内部模块依赖（Mermaid）
## 7. 模块—功能点明细（`## 功能需求` 下 `###` 标题串，可仅列摘要）
## 8. PRD 路径索引
## 9. 开放问题与 PRD 疑似笔误
## 10. 维护说明
## 11. 功能点全量目录与文档×文档细粒度关系（或独立 `PRODUCT_FEATURE_POINT_LEVEL_APPENDIX.md`）
```

### 4.1 附录文件（功能点级，v3 必选）

与主图谱 **同目录** 维护 **`PRODUCT_FEATURE_POINT_LEVEL_APPENDIX.md`**（或等价文件名），至少包含：

| 章节 | 内容 |
|------|------|
| **A** | 功能点 ID：`M-<模块文件夹名>-<子节标识>`（与 **`## 功能需求`** 下 **`###` 可一一对应**）；`ROLLUP` 表示仅继承主文档 §3 模块三元组、无新增跨文档断言 |
| **B** | 大表：每行对应 BRD 中 **`## 功能需求`** 的一个 **`###` 子节**，列含 **关联本仓库 BRD**、**谓词**、简述 |
| **C** | **文档×文档** 有向边表：源/目标文件夹、谓词汇总、**功能级锚点**（如 `M-a-FR03→M-b-FR02` 或 `M-a-<标题slug>→M-b-<标题slug>`） |
| **D** | （可选）31×31 稀疏邻接或等价索引，便于查「谁连谁」 |

**合并规则**：先完成主文档 **§3** 模块三元组，再填附录 B；附录与 §3 冲突时 **以 §3 为准**。

## 5. 与其它指令的关系

- **`feature-list-generation-rules_v1.md`** → **`功能清单.md`**：提供 **L2/L3 域与模块命名**、价值链总览；生成图谱时 **§1 产品域划分** 宜与清单 **§2.2 系统功能结构图谱** 对齐（命名不一致须在 **§9 开放问题** 说明）。
- **`prd-generation_v8.md`**：产出标准 20 章 BRD；若 **`## 业务依赖关系`** 与 **`## 功能需求`** 结构稳定，**模块级/功能级**抽取成本最低。
- **`prd-completeness-check.md`**：单份 BRD 质量门禁；**不替代**本指令的 **跨文档三元组** 与 **功能级边**。
- **`api-doc-generation-rules_v1.md` / `api-doc-review-rules_v1.md`**：接口契约；**不纳入**文档×文档三元组主表，但可在 **§4 外部系统** 或 **§9 开放问题** 引用「某 EXT 系统接口见 `{模块}_接口文档.md`」以方便集成阅读。

### 推荐端到端顺序（与项目约定一致时）

1. **`功能清单.md`** → 2. **按模块 BRD**（`prd-generation_v8.md`）→ 3. **`prd-completeness-check.md`** → 4. **模块接口合订**（`api-doc-generation-rules_v1.md`）→ 5. **`api-doc-review-rules_v1.md`** → 6. **本指令** 生成 **`FEATURE_GRAPH.md`**（及附录）。
