## 用户输入

```text
$ARGUMENTS
```

用户可提供：

- 系统名称、定位一句话（可选）
- 信息源：PRD 目录、代码仓库（菜单/路由）、已有 BRD、产品访谈纪要等

## 1. 目标

1. 产出或更新一份 **`功能清单.md`**（或团队约定文件名），**结构**与本文 **§2 文档骨架**一致。
2. **功能结构优先**：在文首元信息之后、各模块详细说明之前，必须生成 **系统功能结构图谱（目录总览）**，符合 **§1.2** 五级语义与 **§2.2** 呈现要求，使读者先见「全树」再读细节。
3. **详尽度**：对每个业务模块，至少落实到 **列表页（查询/列/操作）**、**详情/表单分区**、**状态与关键分支**（若存在）；不得用「支持维护」等空话替代可枚举项。
4. **可追溯**：重要条目尽量标注信息来源（PRD 章节、菜单路径、代码入口）于文内脚注或「编制说明」表。
5. **可合并迭代**：允许在已有清单上**增量补全**，冲突时以 **P0 产品 PRD** 为准并显式记录差异。

### 1.1 跨项目适用（原独立「通用输出规则」已合并至本文）

- 一级业务域名称、模块划分随行业与产品变化；**不得**因「非某样本项目」而省略 **列表/表单/审批/集成/通用能力** 等章节类型。某类确不存在时，须显式写 **「本系统不涉及」** 并简述原因。
- 文首或「编制说明」可标注系统形态：**后台管理为主** / **含移动办公** / **Open API 为主** / **含大数据报表** 等，便于读者校准详尽度期望。
- 领域拆解可采用 **主价值链（计划→交付→结算…）** 与 **用户任务类型（填报/审批/查询/配置/接口消费）** 双维度，避免遗漏横切能力。

### 1.2 五级功能模型（与结构图谱对应）

撰写 **§2.2 系统功能结构图谱** 时，层级语义须与下列模型对齐（与常见「系统功能树」类 UML 一致：上级对下级为 **一对多 N**；目录层多为 **组合**，功能层多为 **聚合**）：

| 层级 | 名称 | 在《功能清单》中的典型对应 | 关系提示 |
|------|------|---------------------------|----------|
| L1 | **系统功能树** | 整份清单覆盖的系统能力边界（根节点，一句总述即可） | 根 |
| L2 | **一级目录节点** | `##` 业务功能域 / 一级菜单域 | 相对根为强包含（组合） |
| L3 | **二级目录节点** | `###` 功能模块 | 相对一级目录为强包含（组合） |
| L4 | **系统功能** | 模块内能力单元：如「列表页」「详情/新建编辑」「审批」「报表」等独立小节 | 相对二级目录多为聚合 |
| L5 | **子功能** | 可验收原子点：具体查询项、列、按钮、表单项、接口能力等（详细章用列表展开） | 相对系统功能多为聚合 |

图谱中 **不要求** 把 L5 全部展开到树图里（避免过长）；L4 下列出代表性 L5 或写「含：…」摘要即可，**完整枚举**放在图谱之后的正文。

## 2. 文档骨架（必须遵守）

以下 **章节层级与顺序**为模板；**一级域名称**、**模块数量**按实际系统裁剪，但**不得省略**已存在能力域下的模块。

### 2.1 文首元信息

```markdown
# {系统中文名} - 功能清单

> 系统定位：{一句话：端到端价值 / 覆盖范围}
> 技术栈：{后端主要栈} / {前端主要栈}（按实际）

---
```

### 2.2 系统功能结构图谱（目录总览，必须紧接文首元信息之后）

**位置**：放在文首 `---` 与第一个业务域 `##` 标题之间；**先于** 所有「列表页/表单」等详细功能点展开。

**作用**：用一张「全系统功能树」让读者建立 **L1→L5** 心智模型，再进入各模块细则。语义须与 **§1.2** 一致。

**呈现方式**（二选一或同时使用，至少一种）：

1. **Mermaid 图（推荐）**：使用 `flowchart` / `graph`，自上而下或从左到右展示层级；节点文案用中文，边可标注 `N` 表示一对多。若需区分组合/聚合，可在图下用简短文字说明：「L2→L3 为目录组合；L3→L4→L5 为功能向子点的聚合」。
2. **嵌套 Markdown 列表**：用缩进列表展开 **L2（一级目录）→ L3（二级目录）→ L4（系统功能）**；L5 在每条 L4 下用子列表 **摘要**（每条 3～8 个代表性子功能，或写「详见下文 §x.x」）。

**必含内容**：

- 图谱中覆盖后续正文将展开的 **全部** **业务域** `##`（一级目录 / L2）与 **模块** `###`（二级目录 / L3），**不得** 仅在正文写模块而图谱缺项。（`## 系统功能结构图谱` 自身是文档总览章节，**不计入** L2 业务域节点。）
- **横切章节**（如审批中心、权限与系统管理、外部接口、系统集成、通用功能）在树中作为 **L2 或 L3** 节点显式出现（位置与后文章节一致）。
- 图谱后可加 **2～4 行**「读图说明」：编号规则、L4/L5 与正文标题的对应方式。

**Mermaid 模板示例**（生成时替换为真实域/模块/能力名；可增加节点，但勿删层级含义）：

```markdown
## 系统功能结构图谱

> 说明：下列为全系统功能树总览；`N` 表示一对多；子功能完整列表见各模块正文。

\`\`\`mermaid
flowchart TB
  ROOT[系统功能树<br/>{系统中文名}]
  ROOT --> L1A[一级目录: {域1}]
  ROOT --> L1B[一级目录: {域2}]
  L1A --> L2A1[二级目录: {模块1.1}]
  L1A --> L2A2[二级目录: {模块1.2}]
  L2A1 --> F1[系统功能: 列表页]
  L2A1 --> F2[系统功能: 详情与表单]
  F1 --> SF1[子功能摘要]
  F2 --> SF2[子功能摘要]
\`\`\`

- **组合/聚合**：一级→二级→模块目录层为规划结构上的强包含；模块下的「系统功能」及其「子功能」为可拆分的功能聚合，与实现迭代粒度一致。
```

**嵌套列表模板示例**（可与 Mermaid 二选一，或 Mermaid + 本列表择一详列）：

```markdown
## 系统功能结构图谱

- **系统功能树**（根）：{系统中文名} — {一句话边界}
  - **一级目录**：{业务域 A}
    - **二级目录**：{模块 A1}
      - **系统功能**：列表页 — 子功能摘要：{查询维度要点}；{行操作要点}
      - **系统功能**：详情/新建编辑 — 子功能摘要：{表单分区要点}
    - **二级目录**：{模块 A2}
      - …
  - **一级目录**：{业务域 B}
    - …
```

### 2.3 业务功能域（一级标题 `##`）

按 **业务价值链** 或 **产品菜单域** 划分，例如：`## 一、需求管理`、`## 二、派工管理`……

- 命名：**中文 + 业务含义清晰**；编号「一、二、三」或「1 2 3」全文统一。
- 每个一级域下包含若干 **模块**（`###`）。
- **注意**：业务域章节若使用「一、二、三」编号，**系统功能结构图谱**（§2.2）仍须单独使用标题 **`## 系统功能结构图谱`**（或等价中文），**不得** 与第一个业务域合并或省略。

### 2.4 功能模块（二级标题 `###`）

编号建议：`### 1.1 {模块名称}`，与一级域联动（同一域内连续编号）。

每个模块**至少**区分下列能力类型（**无则写「本模块不涉及」**）：

| 能力类型 | 典型标题 | 必须写清的内容 |
|----------|----------|----------------|
| 列表页 | `**列表页功能**` | 查询/筛选维度；列表列；行/工具栏操作 |
| 详情/申请 | `**详情页**` / `**新建/编辑**` | 表单分区（基本信息、明细、附件等） |
| 审批 | `**审批页**` / `**审批中心**` | 待办、通过/驳回、上一条下一条等 |
| 其它 | 备忘录、批量操作、报表等 | 独立小节 |

### 2.5 横切能力（建议独立章节）

在业务模块之后，**必须**单列（若系统存在）：

| 章节 | 内容 |
|------|------|
| **审批中心** | 汇总各审批类型与通用操作（表格） |
| **权限与系统管理** | 权限节点、日志、重试、审批实例等 |
| **外部接口** | 对外提供 API 的分类表（对接方、能力） |
| **系统集成** | 与外部系统的集成方式与数据交换表 |
| **通用功能** | 导入导出、附件、打印、列配置、迁移等 |

### 2.6 文尾

- 修订记录或「最后更新」日期（可选）。
- **编制说明**（推荐）：信息源、缺口、待产品确认项。

## 3. 详尽度规则（核心）

### 3.1 列表页（每个有列表的模块必填）

**查询/筛选**须以**可枚举字段**列出（勿写「多条件查询」止笔）：

- 正确示例：「需求单号、状态、经办部门、项目来源/编号、创建时间、提交时间、业务模式…」
- 错误示例：「支持按条件筛选」。

**列表显示**须列出**主要列**（与产品列设置默认值或核心列一致）：

- 正确示例：「需求单号、状态、需求金额、币种、经办部门、提交人/时间、当前处理人…」
- 若存在 **列设置/自定义列**，须写「列设置（自定义显示列）」。

**操作**须区分：

- **工具栏**：新增、导出、导入、打印、批量删除等
- **行内/单条**：查看、编辑、删除、复制、撤销等
- **异步能力**：异步导出、查看导出结果等

### 3.2 详情页 / 表单（每个有表单的模块必填）

按 **表单分区** 拆成多级列表（与 `功能清单.md` 范本一致）：

- **基本信息**：字段级能力（选择/填写/自动带出/只读展示）
- **明细信息（行级别）**：行类型（分包/租赁等）、行操作（增删改）
- **关联区**：TS 关联、合同、上游单据等
- **附件**：上传、下载、预览、批量下载
- **审批人/评审/沟通**：查看配置、历史记录、沟通增删查
- **表单操作按钮**：保存/提交/撤销/删除等（与状态联动说明）

### 3.3 审批类

- 若存在 **审批中心/待办**：写 **待审批对象类型、通过/驳回、意见、上一条/下一条、查看历史**。
- 若 **仅流程内审批**：在对应模块写清 **触发时机**。

### 3.4 基础数据 / 配置类

- **CRUD** + **导入/导出/同步/日志**（若存在必须写出）。
- **配置项**尽量用**业务名称**枚举（如证书类型、系数表），避免只写「可配置」。

### 3.5 外部接口与系统集成

- **外部接口**：表格列建议包含 **接口分类 / 消费方系统 / 主要能力**。
- **系统集成**：**集成系统 / 方式 / 主要数据交换**；消息队列、调度、检索、文件云等**横切中间件**单列。

### 3.6 通用功能

至少覆盖 **数据操作**（导入导出打印列配置）、**附件**、**审批流相关**（发起、撤回、多级）；若有 **历史数据迁移**，单列迁移对象类型枚举。

## 4. 信息源与优先级

| 优先级 | 来源 | 用法 |
|--------|------|------|
| P0 | 已评审产品 PRD / 需求说明书 | 菜单、字段、状态、操作以产品为准 |
| P1 | 运行系统界面 + 路由/菜单配置 | 补全遗漏按钮与列 |
| P2 | 后端 Controller / 前端路由 / 权限码 | 追溯实现；**禁止**无依据编造业务含义 |
| — | 缺口 | 写入「编制说明」或文内 **待补充**，不得静默省略模块 |

**冲突处理**：PRD 与代码不一致时，**并列记录**并标注待确认方。

---

## 5. 禁止与反模式

- **禁止**用「支持 CRUD」「可维护」等无字段、无操作枚举的描述代替清单。
- **禁止**合并多个无关模块为一个 `###` 导致无法定位功能边界。
- **禁止**省略「导入/导出/异步导出/审批」等横切能力若产品已提供。
- **禁止**清单与对外接口文档、集成列表 **严重矛盾** 而不加说明。
- **禁止**省略 **§2.2 系统功能结构图谱** 或将详细功能点写在图谱之前（读者应先见总树）。

## 6. 输出与文件约定

- **默认文件名**：`功能清单.md`；
- **编码**：UTF-8；Markdown 规范标题层级，避免跳级（`#` → `##` → `###`）。
- **增量更新**：若文件已存在，**合并**新模块并更新文尾日期；删除功能须与产品确认后标注「已下线」或移除。

## 6.1 与下游文档流水线衔接（推荐执行顺序）

本文件产出的 **`功能清单.md`** 可作为后续规范化交付的**拆分清单与验收范围锚点**。推荐在同一项目内按下列顺序执行（规则文件名以仓库 `commands/` 下实际文件为准）：

| 步骤 | 规则 / 产出 | 要点 |
|------|----------------|------|
| 1 | **本文** → `功能清单.md` | 先定 **§2.2 系统功能结构图谱** 与 **L2/L3 模块边界**；横切能力（审批、外部接口等）在清单中显式成节。 |
| 2 | **`prd-generation_v8.md`** | **按模块 `###` 与清单中的「系统功能 / 子功能」逐项**生成 `{功能点名称}_业务需求文档.md`；**`## 模块归属信息`** 中的路径须与清单 **业务域 `##` + 模块 `###` 标题语义**一致（编号可变，业务含义须对齐）。 |
| 3 | **`prd-completeness-check.md`** | 每份 BRD 生成后（或按批次）执行；报告入同目录 `prd-review/`。 |
| 4 | **`api-doc-generation-rules_v1.md`** | **按模块或按「外部接口」章节**合订接口文档；接口范围优先来自 BRD **`## 技术组件清单`** 与清单 **§2.5 外部接口 / 系统集成** 表，避免遗漏或虚构。 |
| 5 | **`api-doc-review-rules_v1.md`** | 对接口合订本 **按接口块**审查；结论与问题清单可回写至接口文档修订记录。 |
| 6 | **`prd-feature-map-generation.md`** | 在 BRD 目录稳定后生成 **三元组图谱** 与功能点附录；用于跨模块一致性与依赖校验。**抽取须与 `prd-generation_v8.md` 标准 20 章一致**（依赖真源 **`## 业务依赖关系`**，功能点真源 **`## 功能需求`** 下 `###`），详见该规则 **§3.0**。 |

**追溯约定（强烈建议写入清单「编制说明」）**：

- 为每个 **`###` 模块** 预留或维护 **「对应 BRD 路径 / 接口文档章节」** 列或脚注（可迭代补全），避免清单与 BRD、OpenAPI 三类资产长期脱节。
- **`功能清单.md`** 中 **外部接口** 表的 **能力简述** 与接口文档 **接口索引** 的 **说明** 列应 **同语义**；冲突时以 **P0 PRD** 为准并记录差异。

**批处理提示**：多模块项目可采用「**一模块：功能清单节选 → 该模块全部 BRD → 该模块接口合订 → 审查**」的内循环，再执行全项目 **图谱**，以降低返工。

## 7. 质量自检（生成后必须执行）

- [ ] 图谱之后才是各模块 **详细功能点**（列表/表单等枚举），顺序正确
- [ ] 每个一级域下 **模块覆盖完整**（对照菜单/PRD 目录，无遗漏域）
- [ ] 每个含列表的模块：**查询维度 + 列表列 + 操作** 已枚举或标「待补充」

- [ ] 审批/外部接口/系统集成/通用能力：**已单独成章或显式写「本系统不涉及」**
- [ ] 无「仅概括无枚举」的占位句；待确认项有 **待补充** 或「编制说明」
- [ ] （若启用 **§6.1 流水线**）编制说明或文尾已注明 **BRD / 接口文档** 的对应关系或待补链（可选，但推荐）

## 8. 参考结构样例（节选）

以下仅示意 **层级与颗粒度**，生成时替换为真实系统内容；**实际输出须在 `---` 后先写 `## 系统功能结构图谱`（§2.2），再写业务域章节**。

```markdown
## 系统功能结构图谱

（此处：Mermaid 全树 或 嵌套列表，覆盖后文所有 `##` / `###`）
```
