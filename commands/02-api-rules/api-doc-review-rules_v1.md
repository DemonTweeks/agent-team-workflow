## 用户输入

```text
$ARGUMENTS
```

用户可提供：接口文档路径、（可选）**`功能清单.md`** 路径与模块名，用于流水线可选对账。

## 目标

- 对**单份或批量**接口文档（Markdown/Word/HTML/纯文本）执行**完整性**与**规范性**检查。
- 以 **「每个接口 = 独立一块（二级标题）」** 为粒度逐项核对；多接口合订时额外检查**索引**与**通用请求头附录**。
- 输出**通过/不通过**结论、**问题清单**（阻断/主要/建议）与**可执行修改建议**。

### 文档流水线中的位置

- **上游**：**`api-doc-generation-rules_v1.md`** 产出接口合订本；依据常来自 **`prd-generation_v8.md`** 的 BRD **`## 技术组件清单`** 与 **`功能清单.md`**（`feature-list-generation-rules_v1.md`）**外部接口** 表。
- **下游**：定稿接口文档可作为 **`prd-feature-map-generation.md`** 中 **EXT:** 系统或集成边的辅助证据；**模块间主依赖真源** 仍以 BRD **`## 业务依赖关系`**（`prd-generation_v8.md` 第 18 章）为主，接口文档不替代该章。

**可选对账**：若用户提供 **`功能清单.md`** 与本接口文档所属模块，在报告末尾可增加 **「与功能清单外部接口表对账」**：清单中的 **接口分类/消费方/能力** 是否在 **接口索引** 中有对应或已标注「不包含在本服务」（遗漏且无说明 → **建议** 或 **主要**）。

## 检查范围

- **必检**：每个接口块是否包含规定的**必选子节**与表格结构；文档级索引与附录是否合理。
- **选检**：与后端实现（Controller、DTO、Swagger）一致性（若用户提供代码或 OpenAPI）。

---

## 一、文档级检查（多接口合订时必检；单接口可跳过不适用项）

| 项 | 要求 | 不满足时 |
|----|------|----------|
| 接口索引 | 存在**目录或索引表**（序号、接口名、method、路径、简述），便于定位各接口块 | **主要** |
| 通用请求头 | 全文仅**一处**完整通用头表，或每块自洽重复（重复时检查是否一致） | 多处冲突 → **主要** |
| 约定说明 | 基地址、网关、认证方式至少其一有说明（或每块内自给自足） | **建议** |

单接口文档：无索引可接受；通用请求头须在**该接口的输入**中出现。

---

## 二、按接口块的完整性检查（每个 `##` 接口块必检）

### 2.1 块边界

- 每个业务接口对应**一个二级标题** `## …`（如 `## 1. xxx`）。
- 块内必选子节（**标题语义一致即可**，允许「1. 接口说明」等形式）：

| 必选子节 | 最低要求 | 缺失时 |
|----------|----------|--------|
| **接口说明** | URL/path、method、功能说明；网关与环境建议有，未知标「待补充」 | **阻断**或**主要** |
| **输入** | 含通用请求头（或指向附录）+ **URL/Path 参数**小节 + **请求体**小节 | **主要** |
| **输出** | 顶层 code/bo/other（或映射说明）+ ServiceCode 结构 + bo 字段说明 | **主要** |
| **调用实例** | 至少 1 组请求+响应，含 method 与路径/Body 关键信息 | **阻断**或**主要** |
| **错误编码** | 错误/业务码表或与输出章交叉引用且**可定位**；禁止仅写「见代码」 | **主要** |
| **调用方** | 调用系统或责任方至少一项实质内容 | **主要** |

### 2.2 输入子节细化

- **URL/Path 参数** 表列齐全：**序号、属性、位置、必填、中文名、类型、说明**（或与模板等价列）。
- **请求消息体** 表列齐全：**序号、属性、必填、中文名、类型、长度、说明**。
- 无 URL 参数或无 Body 时须**显式**写「无」，不得省略小节。

### 2.3 通用请求头

以下 **8** 项须在**附录或本块输入**中出现一次（列含：属性、必填、长度、说明）：

- X-System-Code（N，20）
- X-Auth-Value（Y，64）
- X-Emp-No（Y，20）
- X-Lang-Id（N，10）
- X-Origin-ServiceName（N，50）
- X-Org-Id（N，50）
- X-Tenant-Id（Y，50）
- X-Itp-Value（N，50）

**缺任意属性且无「项目裁剪声明」** → **主要**。

### 2.4 输出子节细化

- 顶层须体现 **code、bo、other** 语义（字段名可映射说明）。
- **ServiceCode** 须含 **code、msgId、msg**（或等价）。
- 标准 **0000～0005** 至少在文档某处说明（块内或全文共一处 + 引用）。

### 2.5 调用实例

- 至少 **1 组**可对照执行的示例。
- 示例中的 path、method 与**接口说明**一致；不一致 → **主要**。

### 2.6 错误编码

- 表含 **业务/服务错误** 与 HTTP 分层或文字区分，**不与** 3.x 成功结构矛盾。
- 仅列 HTTP 无业务码 → **建议**补充。

### 2.7 扩展子节（选检）

若文档包含 **安全性、业务规则、接口依赖、接口约束、性能约束、自动化用例**：

- **安全性**：Token/鉴权/资源码不宜全空；全「待补充」→ **主要**。
- **性能约束**：若用表，表头建议含引用方、并发、响应时间、数据量等维度；允许空数据行。

---

## 三、规范性检查

### 3.1 术语与拼写

- Response、method、URL 全文统一；混用 → **建议**。
- HTTP 方法建议大写。

### 3.2 表格与类型

- 类型列优先 string/int/boolean/object/array 等与契约一致；模糊词 → **建议**。
- 必填列：Y/N/条件必填；整列空 → **主要**。

### 3.3 网关与完整 URL

- 仅相对路径且无网关说明 → **建议**在接口说明或实例中补全示例。

### 3.4 错误码与输出一致性

- 错误编码与 **ServiceCode / bo** 描述矛盾 → **主要**。

---

## 四、与实现一致性（可选）

- [ ] path + method 与 Controller 一致
- [ ] 字段名与 DTO/VO 一致
- [ ] 校验注解与文档必填一致

不一致 → **主要**或**阻断**（视是否已发布契约）。

---

## 五、严重级别

| 级别 | 含义 |
|------|------|
| 阻断 | 无法作为集成契约；缺整节、示例全无、路径错误 |
| 主要 | 缺子节、缺表、缺关键字段、块内不一致 |
| 建议 | 可读性、索引、术语、示例丰富度 |

---

## 六、输出格式（执行本命令时必须）

1. **结论**：通过 / 不通过（列阻断、主要数量）。
2. **文档级检查结果**（第一节）：索引、附录、适用则勾选。
3. **按接口块结果**：每个接口块一节，列出 2.1～2.6 勾选与问题。
4. **规范性小结**：第三节摘要。
5. **问题清单表**：编号、级别、接口块/章节、描述、期望修复。
6. **（可选）与实现差异**：若提供代码。

---

## 七、自检清单（执行结束前勾选）

- [ ] 每个 `##` 接口块已核对 **6 个必选子节**
- [ ] 通用请求头 8 项已在附录或块内覆盖
- [ ] 输入两子表（URL 参数、Body）列完整或显式「无」
- [ ] 输出与错误编码已交叉核对
- [ ] 调用实例与接口说明 URL/method 一致
- [ ] 调用方非空或已标注待补充及责任人
