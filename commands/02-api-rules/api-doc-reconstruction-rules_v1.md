## 用户输入

```text
$ARGUMENTS
```

用户可提供：服务名、模块名、Controller 代码路径、OpenAPI/Swagger 路径、DTO/VO 路径、输出文件名。

## 目标

- 根据后端 Controller 方法定义，**按本模板生成完整接口文档**。
- **每个接口单独成块**（独立二级标题），块内子节顺序、表格列与模板**一致**；缺信息时标注「待补充」并列入待确认项，**不得静默省略必选子节**。
- **按模块组织**：文档章节体现 `模块 -> Controller -> 方法` 三级结构，便于与 `02-设计文档.md` 交叉引用。
- 多接口合订为一份文档时：须有**文档级目录/索引**，**通用请求头**可只写一次（见「多接口合订」）。

### 与 BRD / 功能清单衔接（推荐）

- **输入优先级**：**P0** 已实现契约（Controller 代码、OpenAPI/Swagger、DTO/VO）；**P1** 对应模块 `{功能点}_业务需求文档.md` 中的 `## 技术组件清单`（API 路径、Controller 方法）；**P2** `功能清单.md`（`feature-list-generation-rules_v1.md`）中 **外部接口 / 系统集成** 表所列能力（用于防遗漏、写「调用方」「业务说明」），**不得**仅凭清单无代码编造字段。
- **分批策略**：**按模块或按业务域** 合订（如 `{模块名}_接口文档.md`），便于与 `api-doc-review-rules_v1.md` 逐批审查及与 BRD 目录对照。
- **生成后**：必须用 `api-doc-review-rules_v1.md` 对合订本做 **按接口块** 审查，并修订 **阻断/主要** 项后再定稿。
- **追溯**：文档级「约定与范围」或修订记录中可注明 **覆盖的 BRD 路径 / 功能清单章节**，便于与 `prd-feature-map-generation.md` 及集成方对齐。

## 文档级结构（合订多接口时）

在**第一个具体接口块之前**，必须包含：

| 块 | 内容 |
|----|------|
| 标题 | `# {服务名} 接口文档` |
| 修订记录 | 版本、日期、作者、变更摘要（可选） |
| 服务概览 | 服务职责、技术栈、基地址、网关前缀、认证方式、读者对象 |
| 模块目录 | 模块清单与职责简述，便于快速定位 |
| **通用请求头（附录）** | 全文只保留 **一处** 完整表；各接口块内写「同「通用请求头」」或「见附录 A」 |
| **接口索引** | 表格：**序号、模块、接口中文名、Controller#method、method、路径、说明**（便于跳转） |

单接口仅一页时：可直接进入「接口块」，**通用请求头**放在该接口块的 **输入** 首节。

---

## 模块章节结构（顺序固定）

每个模块使用**一个二级标题**作为边界：

```markdown
## {序号}. {模块名}

> **模块职责**：一句话描述该模块的业务边界与核心实体  
> **路由前缀**：`/api/{module}/**`  
> **Controller 列表**：`XxxController`, `YyyController`

### {序号}.1 模块接口目录

### {序号}.2 接口详情

#### {序号}.2.n {Controller#method} — {接口中文名称}

##### 1. 基本信息

##### 2. 输入

##### 3. 输出

##### 4. 调用实例

##### 5. 错误编码

##### 6. 调用方

##### 7. 调用链摘要（可选）

##### 8. 非功能特征（可选）
```

以上 **6 个子节标题建议字面一致**（允许前缀编号如「1. 基本信息」，但**语义与顺序**须一致）。

---

### 子节 1：基本信息（生成规则）

必须覆盖（表格或列表，信息不得空占位无说明）：

| 子项 | 说明 |
|------|------|
| 模块名 | 所属模块 |
| Controller | 类全名或简名 |
| 方法名 | Java/Go 方法名 |
| HTTP Method | GET / POST / PUT / DELETE / PATCH 等（大写） |
| Path | 相对路径或与网关拼接后的路径模式 |
| 版本 | 接口版本号（如 v1, v2） |
| 鉴权方式 | Token / OAuth / RBAC / 无 |
| 角色要求 | 所需角色或权限码 |
| 租户要求 | 是否多租户隔离 |
| 限流策略 | 限流规则或阈值 |
| 功能说明 | **一句话**业务用途与典型调用场景 |
| 环境说明 | 测试/生产、BFF、域名、VPN、内网限定等 |

---

### 子节 2：输入（生成规则）

#### 2.0 通用请求头

若全文已在**文档附录**定义，此处写：「见《通用请求头》」或「同附录 A」，并补**本接口额外或覆盖**的 Header（无则写「无额外 Header」）。

若本块为全文唯一接口，须列出**完整**通用头（与下表一致或按项目裁剪后**显式声明**）：

| 属性 | 必填 | 长度 | 说明 |
|------|------|------|------|
| X-System-Code | N | 20 | 系统编码 |
| X-Auth-Value | Y | 64 | 认证令牌 |
| X-Emp-No | Y | 20 | 员工编号 |
| X-Lang-Id | N | 10 | 语言标识 |
| X-Origin-ServiceName | N | 50 | 来源服务名 |
| X-Org-Id | N | 50 | 组织标识 |
| X-Tenant-Id | Y | 50 | 租户标识 |
| X-Itp-Value | N | 50 | 内部透传值 |

项目另有强制 Header 时，在表下增加 **补充 Header** 小节。

#### 2.1 Path 参数

表格列：**序号、属性、类型、必填、约束（regex/range）、业务含义、示例**。

无参数时写「无」，并保留表头一行。

#### 2.2 Query 参数

表格列：**序号、属性、类型、必填、默认值、范围、排序/分页语义、兼容说明、示例**。

无参数时写「无」，并保留表头一行。

#### 2.3 Body 参数

- JSON 树结构展示
- 每字段表格列：**序号、属性、类型、必填、默认值、nullable、校验规则、中文名、说明、示例、来源**
- 嵌套对象与数组元素结构必须展开为子表
- GET 且无 Body：写「无请求体」，并说明参数仅 Query/Path

---

### 子节 3：输出（生成规则）

#### 3.1 Response 顶层结构

| 名称 | 类型 | 说明 |
|------|------|------|
| code | ServiceCode 或 string | 状态码 |
| message | string | 状态信息 |
| data / bo | object / array | 业务数据 |
| requestId | string | 请求追踪ID |
| timestamp | long / string | 响应时间戳 |

若实际字段为 `data`/`result`/`bo` 等，在说明列**映射语义**，并保留**真实 JSON 名**。

#### 3.2 状态与消息结构（ServiceCode）

| 名称 | 类型 | 说明 |
|------|------|------|
| code | string | 状态码 |
| msgId | string | 消息类型/ID |
| msg | string | 可读信息 |

**标准 code 含义**（至少说明一次，可与「错误编码」交叉引用）：

| code | 含义 |
|------|------|
| 0000 | 成功 |
| 0001 | 服务器异常 |
| 0002 | 身份校验失败 |
| 0003 | 业务权限不足 |
| 0004 | 参数校验失败 |
| 0005 | 服务程序异常 |

#### 3.3 业务对象（data/bo）

- 表格列出 **data/bo 主要字段**：名称、类型、必填、说明
- 嵌套可分子表展开
- 每字段列：**序号、属性、类型、nullable、中文名、说明、示例**

---

### 子节 4：调用实例（生成规则）

至少 **1 组**「请求 + 响应」：

| 内容 | 要求 |
|------|------|
| 请求 | method、完整 URL 或 path + 网关示例、关键 Header（可脱敏）、Query、Body（JSON 格式化） |
| 响应 | 完整 JSON 或关键字段节选并注明「节选」 |
| cURL（可选） | 便于复制执行 |

失败场景示例可另起一小段（可选）。

---

### 子节 5：错误编码（生成规则）

本块**集中**列出本接口可能返回的错误，避免与「输出」重复时写「同 3.2，下列为补充」。

表格建议列：

| 错误码 / HTTP | MessageId / 内部码 | Message 示例 | 场景说明 | 处理建议 |
|---------------|-------------------|--------------|----------|----------|
| | | | | |

- **分层**：HTTP 状态（4xx/5xx）与 **业务 code**（含 0000～0005 及业务域错误码）分行列出，**禁止**混为一列不加说明。
- 与 **Swagger / 代码** 不一致时标注「待与实现核对」。

---

### 子节 6：调用方（生成规则）

| 子项 | 说明 |
|------|------|
| 调用方系统 / 应用 | 如门户、BFF、批处理、某微服务 |
| 责任角色 | 产品/研发接口人 |
| 联系方式 | 可选：邮箱、群组 |
| 消费说明 | 同步/异步、是否允许重试、幂等期望 |

---

### 子节 7：调用链摘要（可选，建议保留）

| 子项 | 说明 |
|------|------|
| 主链路 | Controller -> Service -> Repository -> DB/External |
| 关键分支 | 条件分支、异步调用 |
| 关键异常 | 主要异常抛出点与处理 |
| 事务边界 | 是否事务、传播行为 |

---

### 子节 8：非功能特征（可选，建议保留）

| 子项 | 说明 |
|------|------|
| 幂等性 | 幂等键、窗口期 |
| 缓存策略 | 读写策略、失效策略 |
| 重试机制 | 重试次数、退避策略 |
| 超时设置 | 连接超时、读取超时 |
| 审计日志 | 是否记录、记录字段 |
| 性能风险 | 大数据量、慢查询提示 |

## 字段描述质量要求

- 禁止「同义重复描述」（如 `status: 状态`）
- 金额字段必须写清币种与精度
- 时间字段必须写清时区与格式
- 标识字段必须写清生成规则和唯一性
- 布尔字段必须描述 `true/false` 业务语义
- 枚举字段必须列出全部可选值及含义

---

## 命名与格式

- **术语**：全文 `Response`、`method`、URL 大小写统一
- **表格**：空单元格用「—」或「待补充」
- **版本**：文档末尾可附修订记录
- **代码引用**：字段名、类名、方法名用反引号包裹

---

## 质量自检（生成后必须执行）

- [ ] 每个接口块均含 **6 个必选子节**（基本信息、输入、输出、调用实例、错误编码、调用方）
- [ ] 输入含 **通用请求头（或指向附录）** + **Path 参数** + **Query 参数** + **Body 参数**
- [ ] 输出含 **顶层结构** + **ServiceCode** + **data/bo 字段说明**
- [ ] 至少 **1 组** 调用实例；错误表与 **code/msg** 不矛盾
- [ ] 多接口时存在 **索引表**，通用请求头 **不重复粘贴**（或仅附录一处）
- [ ] 扩展子节按需齐全，未涉及写「本接口不涉及」
- [ ] 字段描述符合「字段描述质量要求」
- [ ] （若启用 **与 BRD/功能清单衔接**）接口索引与 `## 技术组件清单` / 清单外部接口表 无未解释的重大遗漏（待补充须有责任人或 TODO）
- [ ] 每个接口可追溯到代码证据（Controller 方法签名、DTO 定义）

---

## 增量更新规则

- 变更感知粒度：模块级 + 方法级
- 当以下对象变更时触发重算：
  - Controller 签名/路由
  - DTO/Schema 字段
  - 鉴权与事务注解
- 仅重生成受影响模块章节，不重写全文件

版本策略：
- 章节结构变化 -> `major`
- 模块新增/删除 -> `minor`
- 字段/规则修订 -> `patch`
