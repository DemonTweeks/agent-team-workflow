## 用户输入

```text
$ARGUMENTS
```

用户可提供：`01-接口文档.md` 路径、服务代码路径（Controller/DTO/VO）、（可选）`功能清单.md` 路径与模块名、（可选）`quality-report.json` 路径。

## 目标

- 对**单份或批量**接口返构文档执行**完整性**、**规范性**、**代码一致性**与**证据可追溯性**检查。
- 以 **「每个接口 = 独立一块（四级标题 `####`）」** 为粒度逐项核对；多模块合订时额外检查**模块目录**、**接口索引**与**通用请求头附录**。
- 输出**通过/不通过**结论、**问题清单**（阻断/主要/建议）与**可执行修改建议**。

### 文档流水线中的位置

- **上游**：`api-doc-reconstruction-rules_v1.md` 产出接口文档；依据来自后端代码（Controller、DTO、OpenAPI/Swagger）。
- **下游**：定稿接口文档作为 `design-doc-reconstruction-rules_v1.md` 的交叉引用输入，也可作为 `prd-feature-map-generation.md` 中 EXT: 系统边的辅助证据。
- **可选对账**：若用户提供 `功能清单.md`，在报告末尾增加「与功能清单外部接口表对账」：清单中的接口分类/消费方/能力是否在接口索引中有对应或已标注「不包含在本服务」。

## 一、文档级检查（多模块合订时必检；单模块可跳过不适用项）

| 项 | 要求 | 不满足时 |
|----|------|----------|
| 服务概览 | 存在服务职责、技术栈、基地址、网关前缀、认证方式、读者对象说明 | **主要** |
| 模块目录 | 存在模块清单与职责简述，便于快速定位 | **主要** |
| 接口索引 | 存在表格（序号、模块、接口中文名、Controller#method、method、路径、说明），便于跳转 | **主要** |
| 通用请求头 | 全文仅**一处**完整通用头表（附录），或每块自洽重复（重复时检查是否一致） | 多处冲突 → **主要** |
| 约定说明 | 基地址、网关、认证方式至少其一有说明（或每块内自给自足） | **建议** |
| 修订记录 | 版本、日期、作者、变更摘要（可选但推荐） | 缺失 → **建议** |

单模块文档：无模块目录可接受；通用请求头须在该模块的首个接口块输入中出现。

---

## 二、模块级检查（每个 `##` 模块块必检）

### 2.1 模块边界

- 每个模块对应**一个二级标题** `## {序号}. {模块名}`。
- 模块块内必须包含：

| 必选子节 | 最低要求 | 缺失时 |
|----------|----------|--------|
| 模块说明 | 职责、边界、关键实体 | **主要** |
| 接口目录 | 表格（方法、路径、鉴权、幂等、事务） | **主要** |
| 接口详情 | 至少一个方法级接口块 | **阻断** |

### 2.2 模块接口目录表

表格必须包含：

| 检查列 | 要求 | 缺失时 |
|--------|------|--------|
| 序号 | 连续编号 | **建议** |
| 方法 | Controller#method | **主要** |
| 路径 | HTTP Method + Path | **主要** |
| 鉴权 | 鉴权方式或「无」 | **建议** |
| 幂等 | 是/否/幂等键 | **建议** |
| 事务 | 是/否/传播行为 | **建议** |

---

## 三、按接口块的完整性检查（每个 `####` 接口块必检）

### 3.1 块边界

- 每个业务接口对应**一个四级标题** `#### {序号}. {Controller#method} — {接口中文名称}`。
- 块内必选子节（标题语义一致即可，允许「1. 基本信息」等形式）：

| 必选子节 | 最低要求 | 缺失时 |
|----------|----------|--------|
| **基本信息** | 模块名、Controller、方法名、HTTP Method、Path、版本、功能说明 | **阻断**或**主要** |
| **输入** | 含通用请求头（或指向附录）+ **Path 参数** + **Query 参数** + **Body 参数** | **主要** |
| **输出** | 顶层 code/message/data + ServiceCode 结构 + data/bo 字段说明 | **主要** |
| **调用实例** | 至少 1 组请求+响应，含 method 与路径/Body 关键信息 | **阻断**或**主要** |
| **错误编码** | 错误/业务码表或与输出章交叉引用且**可定位**；禁止仅写「见代码」 | **主要** |
| **调用方** | 调用系统或责任方至少一项实质内容 | **主要** |

### 3.2 基本信息子节细化

| 检查项 | 要求 | 不满足时 |
|--------|------|----------|
| 模块名 | 与上级模块一致 | **主要** |
| Controller | 类全名或简名 | **建议** |
| 方法名 | 与代码一致 | **主要** |
| HTTP Method | GET/POST/PUT/DELETE/PATCH 大写 | **建议** |
| Path | 相对路径或完整路径 | **主要** |
| 版本 | 接口版本号 | **建议** |
| 鉴权方式 | Token/OAuth/RBAC/无 | **建议** |
| 功能说明 | 一句话业务用途，禁止空泛 | **主要** |
| 环境说明 | 测试/生产/BFF/域名等 | **建议** |

### 3.3 输入子节细化

#### 3.3.1 通用请求头

以下 8 项须在附录或本块输入中出现一次（列含：属性、必填、长度、说明）：

- X-System-Code（N，20）
- X-Auth-Value（Y，64）
- X-Emp-No（Y，20）
- X-Lang-Id（N，10）
- X-Origin-ServiceName（N，50）
- X-Org-Id（N，50）
- X-Tenant-Id（Y，50）
- X-Itp-Value（N，50）

**缺任意属性且无「项目裁剪声明」** → **主要**。

#### 3.3.2 Path 参数

表格列必须齐全：**序号、属性、类型、必填、约束、业务含义、示例**。

- 无参数时须显式写「无」，不得省略小节 → **主要**
- 字段名与代码不一致 → **主要**
- 约束未标注（regex/range）→ **建议**

#### 3.3.3 Query 参数

表格列必须齐全：**序号、属性、类型、必填、默认值、范围、排序/分页语义、兼容说明、示例**。

- 无参数时须显式写「无」→ **主要**
- 分页参数未标注（page/size/sort）→ **建议**

#### 3.3.4 Body 参数

- JSON 树结构展示 → **主要**
- 每字段表格列：**序号、属性、类型、必填、默认值、nullable、校验规则、中文名、说明、示例、来源** → **主要**
- 嵌套对象与数组元素结构必须展开为子表 → **主要**
- GET 且无 Body：须写「无请求体」并说明参数仅 Query/Path → **建议**

### 3.4 输出子节细化

#### 3.4.1 Response 顶层结构

| 检查项 | 要求 | 不满足时 |
|--------|------|----------|
| code | 状态码字段 | **主要** |
| message | 状态信息字段 | **建议** |
| data/bo | 业务数据字段 | **主要** |
| requestId | 请求追踪ID | **建议** |
| timestamp | 响应时间戳 | **建议** |

若实际字段为 `data`/`result`/`bo` 等，须在说明列映射语义，并保留真实 JSON 名 → **主要**。

#### 3.4.2 ServiceCode 结构

| 检查项 | 要求 | 不满足时 |
|--------|------|----------|
| code | 状态码 | **主要** |
| msgId | 消息类型/ID | **建议** |
| msg | 可读信息 | **建议** |

标准 code 含义（0000～0005）至少在文档某处说明 → **主要**。

#### 3.4.3 data/bo 字段说明

- 表格列出主要字段：名称、类型、必填、说明 → **主要**
- 嵌套对象分子表展开 → **建议**
- 每字段列：序号、属性、类型、nullable、中文名、说明、示例 → **建议**

### 3.5 调用实例

- 至少 1 组「请求 + 响应」→ **阻断**
- 请求须含：method、完整 URL 或 path + 网关示例、关键 Header、Query、Body（JSON 格式化）→ **主要**
- 响应须含：完整 JSON 或关键字段节选并注明「节选」→ **主要**
- cURL 可选但推荐 → **建议**
- 示例中的 path、method 与基本信息一致；不一致 → **主要**
- 失败场景示例可选 → **建议**

### 3.6 错误编码

- 表含业务/服务错误与 HTTP 分层或文字区分 → **主要**
- 不与输出章的 ServiceCode 矛盾 → **主要**
- 仅列 HTTP 无业务码 → **建议**补充
- 与 Swagger/代码不一致时须标注「待与实现核对」→ **建议**

### 3.7 调用方

| 检查项 | 要求 | 不满足时 |
|--------|------|----------|
| 调用方系统/应用 | 如门户、BFF、批处理、某微服务 | **主要** |
| 责任角色 | 产品/研发接口人 | **建议** |
| 联系方式 | 邮箱、群组 | **建议** |
| 消费说明 | 同步/异步、是否允许重试、幂等期望 | **建议** |

### 3.8 扩展子节（选检）

若文档包含**调用链摘要、非功能特征**：

- **调用链摘要**：主链路 + 关键分支 + 关键异常 + 是否事务 → **建议**
- **非功能特征**：幂等、缓存、重试、超时、审计日志、性能风险 → **建议**
- 全「待补充」且无说明 → **建议**

## 四、字段描述质量检查

| 检查项 | 要求 | 不满足时 |
|--------|------|----------|
| 禁止同义重复 | 如 `status: 状态` → 须写清业务语义 | **主要** |
| 金额字段 | 必须写清币种与精度 | **主要** |
| 时间字段 | 必须写清时区与格式 | **主要** |
| 标识字段 | 必须写清生成规则和唯一性 | **主要** |
| 布尔字段 | 必须描述 true/false 业务语义 | **主要** |
| 枚举字段 | 必须列出全部可选值及含义 | **主要** |

---

## 五、规范性检查

### 5.1 术语与拼写

- Response、method、URL 全文统一；混用 → **建议**
- HTTP 方法大写 → **建议**
- 字段名、类名、方法名用反引号包裹 → **建议**

### 5.2 表格与类型

- 类型列优先 string/int/boolean/object/array 等与契约一致；模糊词 → **建议**
- 必填列：Y/N/条件必填；整列空 → **主要**
- 空单元格用「—」或「待补充」→ **建议**

### 5.3 网关与完整 URL

- 仅相对路径且无网关说明 → **建议**在基本信息或实例中补全示例

### 5.4 错误码与输出一致性

- 错误编码与 ServiceCode / data 描述矛盾 → **主要**

---

## 六、与代码一致性检查（可选但强烈建议）

若用户提供代码或 OpenAPI：

| 检查项 | 要求 | 不一致时 |
|--------|------|----------|
| Path + Method | 与 Controller 注解一致 | **阻断**或**主要** |
| 字段名 | 与 DTO/VO 一致 | **主要** |
| 字段类型 | 与 DTO/VO 一致 | **主要** |
| 必填属性 | 与校验注解（@NotNull/@Valid 等）一致 | **主要** |
| 枚举值 | 与代码枚举定义一致 | **主要** |
| 路由前缀 | 与 @RequestMapping 一致 | **主要** |

---

## 七、证据可追溯性检查

| 检查项 | 要求 | 不满足时 |
|--------|------|----------|
| 证据锚点 | 每个接口至少 1 个证据锚点（file_path + symbol + evidence_type） | **主要** |
| 置信度 | 主文档正文只含 EXTRACTED 结论 | **主要** |
| INFERRED/AMBIGUOUS | 仅出现在「候选/待确认」区块，不得混入正文 | **主要** |
| UNKNOWN_RUNTIME | 未观察到证据的内容必须标记 UNKNOWN_RUNTIME | **建议** |
| 无证据推断 | 禁止无证据推断，否则标记 INVALID_NO_EVIDENCE | **阻断** |

---

## 八、严重级别

| 级别 | 含义 | 示例 |
|------|------|------|
| 阻断 | 无法作为集成契约；缺整节、示例全无、路径错误、无证据推断 | 缺基本信息/输入/输出/调用实例；Controller 方法未覆盖 |
| 主要 | 缺子节、缺表、缺关键字段、块内不一致、字段描述质量不达标、代码不一致 | 缺 Path/Query/Body 参数表；字段名与代码不一致；金额未标币种 |
| 建议 | 可读性、索引、术语、示例丰富度、扩展子节缺失 | 无修订记录；无 cURL；无环境说明 |

---

## 九、输出格式（执行本命令时必须）

1. **结论**：通过 / 不通过（列阻断、主要、建议数量）。
2. **文档级检查结果**（第一节）：服务概览、模块目录、接口索引、通用请求头附录、修订记录。
3. **按模块结果**：每个模块一节，列出模块边界、接口目录检查。
4. **按接口块结果**：每个接口块一节，列出 3.1～3.8 勾选与问题。
5. **字段描述质量小结**：第四节摘要，列不达标字段。
6. **规范性小结**：第五节摘要。
7. **与代码一致性差异**：若提供代码（第六节）。
8. **证据可追溯性检查**：第七节结果。
9. **问题清单表**：编号、级别、模块/接口块/章节、描述、期望修复、证据锚点。
10. **（可选）与功能清单对账**：若提供功能清单。

---

## 十、自检清单（执行结束前勾选）

- [ ] 每个 `##` 模块块已核对模块说明、接口目录、接口详情
- [ ] 每个 `####` 接口块已核对 **6 个必选子节**（基本信息、输入、输出、调用实例、错误编码、调用方）
- [ ] 通用请求头 8 项已在附录或块内覆盖
- [ ] 输入四子节（通用请求头/Path/Query/Body）列完整或显式「无」
- [ ] 输出三子节（顶层结构/ServiceCode/data/bo）完整
- [ ] 字段描述质量检查通过（金额/时间/标识/布尔/枚举）
- [ ] 调用实例与基本信息 URL/method 一致
- [ ] 错误编码与 ServiceCode 不矛盾
- [ ] 调用方非空或已标注待补充及责任人
- [ ] 每个接口至少 1 个证据锚点
- [ ] 主文档正文无 INFERRED/AMBIGUOUS 混入
- [ ] （若提供代码）Path/Method/字段名/类型/必填与代码一致
